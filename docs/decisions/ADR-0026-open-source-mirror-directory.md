# ADR-0026 仓库内开源镜像目录

## 状态

已接受

## 背景

当前 `pg_flashback` 仓库同时包含：

- 开源插件本体源码与 PGXS 回归资产
- 内部研发状态记录、设计计划、实验报告
- 构建产物、日志和临时输出

用户需要把项目整理为可同步到 GitHub 的开源版本，但同时明确要求：

- 根仓库保留内部研发资料
- 开源版本放在仓库内单独目录
- 后续主仓库改动可以稳定同步到该开源目录

## 决策

本轮固定仓库发布结构如下：

- 根仓库继续作为唯一权威研发源
- 新增 `open_source/pg_flashback/` 作为长期维护的开源镜像目录
- 开源镜像不手工长期维护，而是通过 `scripts/sync_open_source.sh` 由白名单规则重建
- 首版开源镜像只保留：
  - `Makefile`
  - `README.md`
  - `pg_flashback.control`
  - `include/`
  - `src/`
  - `sql/`
  - `expected/`
  - 精简后的公开文档
- 首版明确排除：
  - `STATUS.md`
  - `TODO.md`
  - `PROJECT.md`
  - `docs/reports/`
  - `docs/superpowers/`
  - `tests/deep/`
  - `tests/release_gate/`
  - 日志、构建产物、性能采样和其他临时输出

## 为什么不直接手工维护开源子目录

- 人工同步极易漂移
- 很难持续保证“只公开白名单内容”
- 每次发布前都需要重新人工排查脏文件，成本高且不稳定

## 为什么不使用软链接或 subtree

- 软链接不适合作为正式开源仓库根目录
- `git subtree`/历史拆分属于后续仓库治理问题，不是当前最快、最稳的整理路径
- 当前最重要的是先把“可重复得到干净开源目录”的机制固定下来

## 后果

优点：

- 内部研发仓库不需要为开源目录让位
- GitHub 同步对象边界明确
- 每次刷新结果稳定、可重复

代价：

- 根仓库与开源镜像形成双份文件树
- README 与公开文档后续需要持续关注是否漂移
- 对外同步前需要显式执行一次同步脚本

## 运维要求

- 正式对外同步前，统一执行：

```bash
bash scripts/sync_open_source.sh
```

- 同步后检查 `open_source/pg_flashback/` 目录边界，再推送到外部 GitHub 仓库
