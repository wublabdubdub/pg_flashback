# 调试与验证手册

## 先分层，再定位

当前 `pg_flashback` 的问题，优先按这 6 层分：

1. 入口层
2. relation / runtime gate
3. WAL source / scan / index
4. replay
5. TOAST
6. apply / 结果集

不要再按“结果表创建失败 / 结果表内容不对”去分层，当前实现已经没有结果表主链。

## 本机最短验证路径

本机 PG18：

```bash
su - 18pg
psql postgres
```

构建安装：

```bash
make clean
make install
```

全量回归：

```bash
su - 18pg -c 'cd /walstorage/pg_flashback && make installcheck'
```

## 入口层怎么查

先看：

- `sql/pg_flashback--0.1.0.sql`
- `src/fb_entry.c`
- `sql/fb_user_surface.sql`

重点确认：

- 安装面是否只有 `pg_flashback(anyelement, text)`
- 是否仍有人误用旧的 `pg_flashback(text, text, text)`
- 第一个参数是否写成了 `NULL::schema.table`

典型报错：

- `pg_flashback target must be a table row type`
- `pg_flashback must be called in a set-returning context`

## runtime gate 怎么查

先看：

- `src/fb_catalog.c`
- `src/fb_runtime.c`
- `src/fb_guc.c`
- `sql/fb_runtime_gate.sql`

典型报错：

- `target timestamp is in the future`
- `neither pg_flashback.archive_dest nor legacy pg_flashback.archive_dir is set`
- `... contains no WAL segments`
- `fb does not support system catalogs`

## WAL 层怎么查

先看：

- `src/fb_wal.c`
- `src/fb_ckwal.c`
- `docs/architecture/wal-source-resolution.md`

先确认 4 件事：

1. `archive_dest` 是否有效
2. `pg_wal` 是否只是承接 recent tail
3. 错配段是否进入 `recovered_wal`
4. `unsafe window` 是否被正确触发

典型报错：

- `WAL not complete`
- `no checkpoint before target timestamp`
- `fb does not support WAL windows containing ... operations`

## replay 层怎么查

先看：

- `src/fb_replay.c`
- `src/fb_reverse_ops.c`

重点问题：

- `missing FPI`
- 某类 heap WAL record 没有被正确重放
- `ForwardOp` 行像不完整

## TOAST 怎么查

先看：

- `src/fb_toast.c`
- `src/fb_replay.c`

重点问题：

- 历史 chunk 丢失
- finalize 后仍残留 external datum
- 当前值与历史值串用

## apply / 结果集怎么查

先看：

- `src/fb_apply.c`
- `src/fb_apply_keyed.c`
- `src/fb_apply_bag.c`

当前要点：

- keyed 不再构造整表 hash
- bag 不再构造整表 bag
- `pg_flashback()` 通过 SRF 逐行返回

如果出现：

- 空结果异常
- residual 行缺失
- 宽表排序/返回时崩溃

优先检查：

- `heap_copy_tuple_as_datum()` 返回边界
- keyed residual 路径
- bag delta 收敛逻辑

## 进度相关怎么查

先看：

- `src/fb_progress.c`
- `sql/fb_progress.sql`

当前 stage `9` 是：

- `emitting residual historical rows`

不是旧的“materializing result table”。

至少做到：

1. `make clean && make install`
2. `make installcheck`
3. 若改了 TOAST 或 deep 相关逻辑，再补对应 deep 脚本验证

没有这些验证，不要宣称“已经完成”。
