# 调试与验证手册

本文按当前代码现状整理调试与验证路径，默认你面对的是：

- 直接查询型 `pg_flashback(anyelement, text)`
- 当前主链 `checkpoint + RecordRef + block redo + ReverseOpSource + streaming apply`
- 已引入 `archive_dest + archive_command autodiscovery + pg_wal + recovered_wal`

不再按“结果表创建失败 / 结果表内容不对”的旧模型组织。

## 一、先分层，再定位

当前问题优先按这 7 层分：

1. 入口与调用形态
2. relation / runtime gate
3. archive 解析与 WAL 来源
4. WAL scan / RecordRef / unsafe window
5. replay / page baseline / heap redo
6. TOAST 历史重建
7. apply / SRF 发射

绝大多数问题，一旦分层正确，定位会快很多。

## 二、本机最短验证路径

根据仓库约定，本机 PG18 登录方式固定为：

```bash

```

如果改了头文件或不确定依赖是否干净，先完整重装：

## 四、入口层怎么查

先看：

- `sql/pg_flashback--0.1.0.sql`
- `src/fb_entry.c`
- `sql/fb_user_surface.sql`

重点确认：

1. 安装面是否仍只有 `pg_flashback(anyelement, text)`
2. 有没有意外把 debug 函数装进公开 SQL
3. 第一个参数是不是必须写成 `NULL::schema.table`
4. 当前执行器是否给了 `SFRM_ValuePerCall` 或 `SFRM_Materialize`

典型报错：

- `pg_flashback target must be a table row type`
- `pg_flashback must be called in a set-returning context`
- `pg_flashback requires value-per-call SRF mode`

如果问题发生在 SRF 生命周期、提前退出、异常 cleanup，优先看：

- `fb_flashback_cleanup_callback()`
- `fb_unregister_flashback_cleanup()`

## 五、relation / runtime gate 怎么查

先看：

- `src/fb_catalog.c`
- `src/fb_guc.c`
- `src/fb_runtime.c`
- `sql/fb_runtime_gate.sql`
- `sql/fb_relation_unsupported.sql`

重点确认：

1. 目标 relation 是不是普通持久化 heap 表
2. 有没有误把 partitioned parent / matview / unlogged 放过去
3. runtime 目录是否已自动初始化
4. `archive_dest` / `archive_dir` / autodiscovery 到底解析成了什么

典型报错：

- `target timestamp is in the future`
- `... target must be a table row type`
- `fb does not support ...`
- archive autodiscovery 相关错误

## 六、archive 与来源解析怎么查

先看：

- `src/fb_guc.c`
- `src/fb_wal.c`
- `src/fb_ckwal.c`
- `docs/architecture/wal-source-resolution.md`

可以先用回归专用 debug 函数确认 archive 解析结果：

- `fb_archive_resolve_debug()`

相关回归：

- `sql/fb_wal_source_policy.sql`

优先确认 5 件事：

1. 当前到底走的是 `archive_dest`、`archive_dir` 还是 `archive_command`
2. `archive_command` 是否属于当前支持的本地模式
3. `pg_wal` 是否只在补 recent tail
4. 错配段是否进入 `recovered_wal`
5. 三路来源是否能拼出连续 segment 集

典型报错：

- `contains no WAL segments`
- `WAL not complete`
- autodiscovery refusal / explicit configuration required

## 七、WAL scan / RecordRef 怎么查

先看：

- `src/fb_wal.c`
- `include/fb_wal.h`
- `sql/fb_recordref.sql`
- `sql/fb_wal_sidecar.sql`

回归专用 debug：

- `fb_recordref_debug(regclass, timestamptz)`
- `fb_wal_sidecar_debug(regclass, timestamptz)`

它们当前可以帮助回答：

1. anchor 是否找到
2. unsafe 是否触发
3. kept / target_dml / commits / aborts 数量对不对
4. tail inline 是否启用
5. checkpoint sidecar / anchor hint 是否命中

如果这里就已经不对，不要急着看 replay。

## 八、unsafe window 怎么查

当查询直接被拒绝时，先看 detail。

入口层当前会把这些信息带出来：

- `scope=main|toast`
- `operation=...`
- `target=...`
- `toast=...`
- `xid=...`
- `commit_ts=...`
- `lsn=...`

定位顺序：

1. 先确认是主表还是 TOAST
2. 再确认是 truncate / rewrite / storage change 哪一类
3. 最后回到对应时间点和事务复盘

这里最容易误判的点：

- standalone `standby_lock` 当前不再单独阻塞
- TOAST `SMGR TRUNCATE` 当前允许继续 replay，最终缺 chunk 再报

## 九、replay 层怎么查

先看：

- `src/fb_replay.c`
- `include/fb_replay.h`

重点问题类型：

1. `missing FPI`
2. block ready / baseline 缺失
3. `failed to replay heap insert`
4. `invalid old/new offnum during heap update redo`
5. `heap update record missing new tuple data`

调试顺序建议：

1. 先确认 `RecordRef` 是否齐全
2. 再确认 block image / data / main_data 是否被正确抓到
3. 再确认 discover/warm/final 是否在错误页态上执行
4. 最后再看 row image 是否错误

当前一个关键经验：

- aborted record 逻辑上可不进结果，但物理页 replay 不能随便跳过

## 十、TOAST 怎么查

先看：

- `src/fb_toast.c`
- `src/fb_replay.c`
- `sql/fb_toast_flashback.sql`
- `sql/fb_flashback_toast_storage_boundary.sql`

当前常见问题：

1. TOAST chunk 缺失
2. 历史 chunk 在 store 中存在，但 spill 后读不回来
3. TOAST truncate 后历史值仍能否重建
4. 主表 row image 里仍残留 external datum

重点函数：

- `fb_toast_store_sync_page()`
- `fb_toast_store_put_tuple()`
- `fb_toast_store_remove_tuple()`
- `fb_toast_reconstruct_datum()`
- `fb_toast_rewrite_tuple()`

要记住当前行为：

- 历史 store 缺 chunk 时，会先尝试 live fallback
- live 也不行才直接报错
- 不会静默返回残缺值

## 十一、apply / 结果集怎么查

先看：

- `src/fb_apply.c`
- `src/fb_apply_keyed.c`
- `src/fb_apply_bag.c`

当前主线要点：

- keyed 只维护变化 key
- bag 只维护变化 row identity 的 delta
- 扫描结束后还有 residual 阶段
- stage 9 当前表示 residual historical rows 发射

典型问题：

1. 结果行数不对
2. residual 行缺失
3. keyed 替换错版本
4. bag 重复次数不对
5. 宽表 / TOAST-heavy 返回时崩溃

优先检查：

- `fb_apply_finish_scan_phase()`
- keyed 的 `current_seen` / `replacement_tuple`
- bag 的 `delta` / `residual_repeat`
- `fb_apply_emit_as_datum()`

## 十二、进度显示怎么查

先看：

- `src/fb_progress.c`
- `sql/fb_progress.sql`

当前 9 段定义已经和旧模型不同：

1. validate
2. prepare wal
3. build record index
4. replay discover
5. replay warm
6. replay final
7. build reverse ops
8. apply
9. residual 历史行发射

如果看到 stage 9 还被写成“结果表物化”，那是旧口径。

## 十四、修改完成前的最低验证要求

文档、代码、SQL 任一改动，只要你打算声称“完成”，至少要有对应验证证据。

最低建议：

1. `make clean && make PG_CONFIG=/home/18pg/local/bin/pg_config install`
2. `PGPORT=5832 PGUSER=18pg make PG_CONFIG=/home/18pg/local/bin/pg_config installcheck`
3. 若涉及 TOAST / deep / archive 解析，再补对应专项验证

如果这三步没做完，不要宣称“已验证通过”。

## 十五、当前最常见的排障顺序

### 用户说“直接报错”

先查：

1. 入口调用形态
2. relation gate
3. archive 解析
4. unsafe detail

### 用户说“某个 target_ts 结果不对”

先查：

1. `fb_recordref_debug`
2. replay 是否命中 missing baseline / offnum 问题
3. TOAST 是否串用了 live 值
4. keyed/bag residual 是否收敛正确

### 用户说“很慢 / 很占内存”

先分：

1. WAL/resolver 前置扫描慢
2. replay block store 内存高
3. reverse-op source/run spill 高
4. apply / TOAST flatten 热点

不要一上来就怪 apply。

## 十六、最短结论

当前调试的核心不是“多打日志”，而是先判断问题属于哪一层。
只要层级分对，当前代码的边界其实很清楚：入口、来源、WAL、replay、TOAST、apply 各有清晰收口点。
