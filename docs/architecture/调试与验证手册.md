# 调试与验证手册

## 目的

这份手册解决的是维护期最实际的问题：

- 一个错误出现后，先定位在哪一层
- 该跑哪组测试，不要全仓乱跑
- 哪些目录和运行时产物值得看
- 什么错误是“当前代码真有 bug”，什么只是“当前边界本来就不支持”

建议把它和《源码级维护手册》一起看。

## 一张排障分层图

遇到问题时，先把故障分到这 6 层之一：

1. 入口层
   - 参数解析
   - 结果表创建
   - 公开 SQL surface
2. relation / runtime gate
   - relation 不支持
   - archive 配置错误
   - future target
3. WAL source / scan / index
   - `WAL not complete`
   - `no checkpoint before target timestamp`
   - `unsafe=truncate/rewrite/storage_change`
4. replay
   - `missing FPI`
   - `heap insert/update/delete redo` 失败
   - block data / image 缺失
5. TOAST
   - `missing toast chunk`
   - finalize 后仍残留 external datum
6. apply / 结果层
   - keyed 或 bag 结果错误
   - 结果行数不对
   - 结果表内容不对，但前面阶段没有报错

先分层，再进代码。不要一上来就全局搜索。

## 最短复现流程

### 1. 先确认当前环境

本机 PG18 登录方式固定：

```bash
su - 18pg
psql postgres
```

工作区构建默认 PG18：

```bash
make
make install
```

如果你改了 `include/*.h`：

```bash
make clean
make install
```

### 2. 先跑最小回归，不要直接跑 deep

全量回归：

```bash
su - 18pg -c 'PGPORT=5832 psql postgres -Atqc "drop database if exists contrib_regression;" && cd /walstorage/pg_flashback && rm -rf results regression.out regression.diffs && make installcheck'
```

只跑某一条回归时，优先改 `Makefile` 的 `REGRESS` 或手工执行对应 SQL 夹具逻辑，而不是先上 deep。

### 3. 只有这几类问题再进 deep

当问题属于下面这些场景时，再跑 `tests/deep/`：

- 大 WAL 窗口
- batch B 缺页基线
- baseline 快照恢复
- archive / fake `pg_wal` / overlap
- TOAST 规模化
- full 模式续跑

## 建议的排障顺序

每次都按这个顺序走：

1. 先确认它是哪个层级的问题
2. 找最小能复现它的回归或 deep 用例
3. 先读对应文件，再下断点或加日志
4. 修完先复跑最小用例
5. 再复跑同层相关用例
6. 最后看是否需要更大范围回归

## 入口层问题怎么查

症状通常包括：

- 结果表名冲突
- 输入时间文本解析失败
- 公开 SQL surface 与文档不一致
- 扩展安装后找不到函数

先看：

- `sql/pg_flashback--0.1.0.sql`
- `src/fb_entry.c`
- `sql/pg_flashback.sql`
- `sql/fb_user_surface.sql`

关键函数：

- `pg_flashback()`
- `fb_require_result_name()`
- `fb_parse_target_ts_text()`
- `fb_build_create_table_sql()`
- `fb_build_flashback_reverse_ops()`
- `fb_parallel_apply_reverse_ops()`

典型问题判断：

- 如果连 `CREATE EXTENSION pg_flashback` 后函数都不存在，先看安装 SQL，不要先看 C
- 如果函数存在但报“结果表已存在”，看 `pg_flashback()` 里结果表名检查
- 如果返回表创建成功但内容不对，问题已经不在入口层

## relation / runtime gate 怎么查

### 对应代码

- `src/fb_catalog.c`
- `src/fb_guc.c`
- `src/fb_runtime.c`
- `sql/fb_relation_gate.sql`
- `sql/fb_relation_unsupported.sql`
- `sql/fb_runtime_gate.sql`

### 常见报错与判断

`fb does not support temporary relations`

- 看 `fb_catalog_require_supported_relation()`
- 这是产品边界，不是 bug

`neither pg_flashback.archive_dest nor legacy pg_flashback.archive_dir is set`

- 看 `fb_require_archive_dir()`
- 先确认测试脚本是否确实设置了 GUC

`target timestamp is in the future`

- 看 `fb_require_target_ts_not_future()`
- 先确认测试使用的是 `clock_timestamp()` 之后还是之前

`pg_flashback ... directory is not writable`

- 看 `fb_runtime_ensure_initialized()`
- 问题在 runtime 目录权限，不在 WAL

### 手工快速检查 SQL

```sql
CREATE EXTENSION IF NOT EXISTS pg_flashback;
SELECT fb_check_relation('public.some_table'::regclass);
SHOW pg_flashback.archive_dest;
SHOW pg_flashback.archive_dir;
SHOW pg_flashback.memory_limit_kb;
SHOW pg_flashback.parallel_segment_scan;
```

## WAL source / scan / index 怎么查

### 你应该先看哪些文件

- `src/fb_wal.c`
- `src/fb_ckwal.c`
- `src/fb_guc.c`
- `docs/architecture/wal-source-resolution.md`
- `sql/fb_flashback_storage_boundary.sql`
- `tests/deep/bin/run_batch_d.sh`

### 当前 resolver 规则要背熟

- `archive_dest` 优先
- `pg_wal` 只补 recent tail
- `pg_wal` 错配先转 `recovered_wal`
- 三路都找不到才报 `WAL not complete`

如果你连这条规则都没先确认，后面大概率会误判。

### 常见报错

`WAL not complete`

优先检查：

1. `archive_dest` 是否真有该 segment
2. `pg_wal` 是否被 recycle / mismatch
3. `DataDir/pg_flashback/recovered_wal/` 里有没有转换后的段
4. `fb_ckwal` 是否被调用

代码入口：

- `fb_collect_archive_segments()`
- `fb_validate_segment_identity()`
- `fb_try_ckwal_segment()`
- `fb_convert_mismatched_pg_wal_entry()`

`no checkpoint before target timestamp`

- 说明 `fb_note_checkpoint_record()` 没找到 `target_ts` 之前的 checkpoint
- 先确认 target 选得是不是过早
- 再确认扫描窗口是不是被 prefilter 误缩

`fb does not support WAL windows containing storage_change operations`

- 先看是不是测试本身就触发了 `TRUNCATE` / rewrite / storage change
- 再确认是不是把 checkpoint 回看区间里、发生在 `target_ts` 之前的记录误判成窗口风险

### 运行时产物怎么用

重点目录：

- `DataDir/pg_flashback/meta/`
  - segment prefilter 的 meta cache
- `DataDir/pg_flashback/recovered_wal/`
  - mismatch / recovered 段

若 resolver 异常：

- 先清理 `meta/` 再复现，排除旧 prefilter cache 干扰
- 看 `recovered_wal/` 是否生成了你预期的 segment 名

## replay 层怎么查

### 对应代码

- `src/fb_replay.c`
- `include/fb_replay.h`
- `include/fb_wal.h`

### 你要先建立的判断

replay 问题通常不是“某条 SQL 拼错”，而是三类之一：

1. 页基线没建立起来
2. redo handler 没覆盖或覆盖不完整
3. old/new tuple 抽取时点不对

### 常见报错

`missing FPI for block ...`

第一优先怀疑：

- 该块在 checkpoint 后首条相关记录并不带 image / init
- 当前共享回扫仍没找到更早 anchor

代码入口：

- `fb_replay_note_missing_block()`
- `fb_replay_resolve_missing_anchors()`
- `fb_replay_warm_store()`
- `fb_replay_missing_fpi_error()`

这类问题优先去看：

- `tests/deep/bin/run_batch_b.sh`
- `docs/reports/2026-03-23-deep-pilot-report.md`

`heap insert record missing block data`

- 看 `fb_replay_heap_insert()`
- 先判断是不是 image-only 情况被误当成需要 block data

`heap update record missing new tuple data`

- 看 `fb_replay_heap_update()`
- 重点检查 new block / old block 的 block data 来源切换逻辑

`failed to locate tuple for heap delete/update redo`

- 页状态已经不对了
- 继续查 apply 没意义
- 要回到前序 replay 和 block state

### 建议断点

如果用 `gdb`，这几个点最有价值：

- `fb_replay_execute_internal`
- `fb_replay_run_pass`
- `fb_replay_ensure_block_ready`
- `fb_replay_missing_fpi_error`
- `fb_replay_heap_insert`
- `fb_replay_heap_delete`
- `fb_replay_heap_update`

### 排查策略

如果是 replay 层 wrong result：

1. 先确认 `RecordRef` 是否收到了对应 record
2. 再确认 block 是否被 materialize
3. 再确认 old tuple 是在 redo 前拷、new tuple 是在 redo 后拷
4. 最后才看 reverse/apply

## TOAST 层怎么查

### 对应代码

- `src/fb_toast.c`
- `src/fb_replay.c`
- `sql/fb_toast_flashback.sql`
- `tests/deep/sql/80_toast_scale.sql`

### 当前实现口径

- TOAST relation record 进入同一套 `RecordRef + replay`
- chunk store 分 `live` / `retired`
- retired chunk 会 spill 到 runtime 文件
- finalize 时把 external datum rewrite 成 inline datum

### 常见报错

`missing toast chunk`

优先检查：

1. 是不是历史 chunk 真缺了
2. 还是 replay 只覆盖到部分 touched toast block
3. live fallback 是否本应可用但未命中

关键函数：

- `fb_toast_store_put_tuple()`
- `fb_toast_store_remove_tuple()`
- `fb_toast_reconstruct_datum()`
- `fb_toast_fetch_live_datum()`
- `fb_forward_stream_finalize()`

`unresolved external toast datum after finalize`

- 说明 row finalize 完之后 tuple 里还残留 external pointer
- 问题不在 apply
- 先看 `fb_toast_rewrite_tuple()` 和 `fb_forward_stream_finalize()`

### 运行时文件

TOAST retired spill 文件会落到：

- `DataDir/pg_flashback/runtime/toast-retired-*.bin`

如果你怀疑 retired chunk 逻辑有问题，先确认这些文件有没有生成，以及复现结束后是否被清理。

### 什么时候跑哪组测试

小改动：

- `sql/fb_toast_flashback.sql`

历史值或 finalize 改动：

- `bash tests/deep/bin/run_toast_scale.sh`

更大范围：

- `bash tests/deep/bin/bootstrap_env.sh --full`
- `bash tests/deep/bin/run_toast_scale.sh --full`

注意：每次完成 TOAST 测试后，要手动清空 `/isoTest/18waldata`。

## apply 层和结果错误怎么查

### keyed 错误

先看：

- `src/fb_apply_keyed.c`
- `sql/fb_flashback_keyed.sql`

常见表现：

- 某 key 对应结果没回退
- 主键变更语义不完整
- `REPLACE` 后结果集残留新 key

判断方法：

- 如果 `ForwardOp` 已经对，但 keyed 结果不对，就是 apply 层
- 如果 `ForwardOp` 自身 old/new 就错，不要继续查 keyed

### bag 错误

先看：

- `src/fb_apply_bag.c`
- `sql/fb_flashback_bag.sql`

常见表现：

- duplicate count 不对
- `REPLACE` 之后计数错
- bag 结果内容对，但重复次数不对

bag 层排障关键：

- 它看的是 `row_identity -> count`
- 它不保物理身份
- 因此“重复行哪一条消失了”不是 bug 判断标准；“重复次数不对”才是

## memory 问题怎么查

### 当前 memory limit 语义

`pg_flashback.memory_limit_kb` 只限制扩展内部已接入跟踪的结构，不等于整个 backend RSS。

当前主要入口：

- `fb_memory_charge_bytes()`
- `fb_index_charge_bytes()`
- `fb_replay_charge_bytes()`
- apply 层各自的 charge 函数

### 常见报错

`pg_flashback memory limit exceeded while tracking ...`

先看报错里的 `what`：

- `RecordRef array`
- `main data`
- `block data`
- `FPI image`
- `BlockReplayStore`
- `ForwardOp array`
- `ReverseOp array`
- `apply keyed row state`
- `apply bag row state`

不同 `what` 对应不同层，不要混看。

### 对应用例

- `sql/fb_memory_limit.sql`

如果你改了 memory accounting，至少复跑这条回归。

## deep 测试什么时候用

### pilot

适合：

- 验证 batch A-E 主链
- 快速判断当前改动有没有破坏已打通路径

命令：

```bash
bash tests/deep/bin/run_all_deep_tests.sh --pilot
```

### full

适合：

- baseline 快照恢复
- 单轮大 WAL
- 状态文件续跑
- 空间守卫

命令：

```bash
bash tests/deep/bin/run_all_deep_tests.sh --full
```

注意：

- full 失败后会保留 baseline snapshot 和状态文件
- 不要失败后立刻把现场清掉，先看 `tests/deep/output/full_state.env`

## 常用运行时检查命令

### 1. 看当前 WAL 源相关设置

```sql
SHOW pg_flashback.archive_dest;
SHOW pg_flashback.archive_dir;
SHOW pg_flashback.debug_pg_wal_dir;
SHOW pg_flashback.parallel_segment_scan;
SHOW pg_flashback.memory_limit_kb;
SHOW pg_flashback.show_progress;
SHOW pg_flashback.parallel_apply_workers;
```

### 2. 最小手工 flashback

```sql
CREATE EXTENSION IF NOT EXISTS pg_flashback;
SET pg_flashback.show_progress = on;
SELECT fb_check_relation('public.t1'::regclass);
SELECT pg_flashback('fb1', 'public.t1', '2026-03-23 08:09:30+08');
SELECT relpersistence FROM pg_class WHERE oid = 'fb1'::regclass;
SELECT * FROM fb1;
```

并行路径最小手工验证：

```sql
SET pg_flashback.parallel_apply_workers = 2;
BEGIN;
SELECT pg_flashback('fb_parallel', 'public.t1', '2026-03-23 08:09:30+08');
ROLLBACK;
SELECT to_regclass('fb_parallel');
```

如果最后还能查到 `fb_parallel`，说明你已经进入了“独立 worker 自主事务创建并填充结果表”的并行路径。

如果你不想让 `NOTICE` 刷屏：

```sql
SET pg_flashback.show_progress = off;
```

如果 `pg_flashback.show_progress = on` 但客户端仍看不到阶段输出，先检查：

- `SHOW pg_flashback.show_progress;`
- `SHOW client_min_messages;`

`client_min_messages` 至少要允许 `NOTICE` 显示。

### 3. 看运行时目录

```bash
su - 18pg
psql postgres -Atqc "show data_directory"
ls -R <DataDir>/pg_flashback
```

## 修改后最小验证策略

不要每次都直接全量回归。按改动层次选：

- 只改入口或安装面：
  - `fb_smoke`
  - `pg_flashback`
  - `fb_user_surface`
- 只改 relation gate：
  - `fb_relation_gate`
  - `fb_relation_unsupported`
- 只改 runtime / resolver：
  - `fb_runtime_gate`
  - `fb_flashback_storage_boundary`
  - 必要时 batch D
- 只改 replay：
  - `fb_flashback_keyed`
  - `fb_flashback_bag`
  - 必要时 batch B
- 只改 TOAST：
  - `fb_toast_flashback`
  - `run_toast_scale.sh`
- 只改 memory：
  - `fb_memory_limit`

再强调一次：先跑最小相关，再扩大范围。

## 当前已知“不必误判成新 bug”的边界

这些问题当前仓库已经明确还没完全收敛：

- batch B / residual `missing FPI`
- `fb_export_undo`
- TOAST store 全量内存硬上限
- 主键变更等剩余正确性
- `parallel_segment_scan` 主路径端到端收益

看到这些领域的异常，先对照 `STATUS.md` / `TODO.md` 判断是不是已知在途项。

## 最后给维护者的一个实战原则

如果结果不对，永远先问自己一句：

“错的是输入事实、页状态、逻辑 op，还是最终应用？”

对应到代码就是：

- `fb_wal.c`
- `fb_replay.c`
- `fb_reverse_ops.c`
- `fb_apply_*`

这四层只要分对了，排障速度会快很多。
