# 核心入口源码导读

本文只讲当前真实入口：

```sql
SELECT *
FROM pg_flashback(
  NULL::public.t1,
  '2026-03-28 10:00:00+08'
);
```

对应代码：

- `src/fb_entry.c`
- `sql/pg_flashback--0.1.0.sql`

不再覆盖已经删除的：

- `pg_flashback(text, text, text)`
- 结果表物化后再查
- 公开安装面的 `fb_flashback_materialize(...)`

## 一、先看入口状态

当前 `src/fb_entry.c` 里既有正式入口，也有几个仅供回归或未来功能占位的 C 入口：

- `fb_version`
- `fb_check_relation`
- `fb_archive_resolve_debug`
- `fb_recordref_debug`
- `fb_wal_sidecar_debug`
- `pg_flashback`
- `fb_export_undo`

但真正安装到公开 SQL 面的，当前只有：

- `fb_version()`
- `fb_check_relation(regclass)`
- `pg_flashback(anyelement, text)`

这一点必须和“源码里声明了哪些 PG_FUNCTION_INFO_V1”区分开。

## 二、`pg_flashback()` 入口的第一件事是什么

### 1. 区分两条入口路径

当前代码里，用户写 `FROM pg_flashback(...)` 时不一定直接进入 `pg_flashback()` 这个 SQL SRF。

已经落地的两条路径是：

- 普通 SQL SRF 路径：`src/fb_entry.c` 中 `pg_flashback()`
- `FROM` 子句路径：`src/fb_custom_scan.c` 中 `FbFlashbackScan`

其中 `FROM pg_flashback(...)` 的 planner 侧会先通过 `fb_pg_flashback_support(internal)` 让模块在规划阶段加载，再由 `fb_custom_scan` 把默认 `FunctionScan` 替换成 `CustomScan`。

### 2. 在 SQL SRF 路径里检查执行模式

`pg_flashback()` 一上来先拿：

- `ReturnSetInfo *rsinfo`

然后根据执行器允许的模式决定内部走法：

- 当前要求 `SFRM_ValuePerCall`

这说明当前代码已经明确收口为：

- 对用户接口仍是同一个 `SETOF anyelement`
- 内部不再切到 materialized 分支

相关代码：

- `src/fb_entry.c` 中 `pg_flashback()`

## 三、为什么第一个参数必须写成 `NULL::schema.table`

关键函数：

- `fb_resolve_target_type_relid()`

它不是解析字符串表名，而是：

1. 取第一个参数在表达式树里已经确定的类型 Oid
2. 调 `typeidTypeRelid(type_oid)` 反推出 relation Oid

因此正确调用形态必须是：

```sql
NULL::public.t1
```

而不是：

- `'public.t1'`
- `'public.t1'::regclass`
- `AS t(...)`

如果拿不到 row type，对应错误就是：

```text
pg_flashback target must be a table row type
```

## 四、目标时间怎么解析

关键函数：

- `fb_parse_target_ts_text()`

当前仍接受第二个参数为 `text`，内部直接调用 PostgreSQL 的：

- `timestamptz_in`

也就是说，时间格式遵循 PostgreSQL 自己的 `timestamptz` 输入规则，不是扩展单独写了一套解析器。

## 五、入口层的 gate 做了什么

关键函数：

- `fb_validate_flashback_target()`

它内部串起：

1. `fb_progress_enter_stage(FB_PROGRESS_STAGE_VALIDATE, NULL)`
2. `fb_require_target_ts_not_future(target_ts)`
3. `fb_require_archive_dir()`
4. `fb_require_supported_target_relation(relid, info_out)`
5. `fb_relation_tupledesc(relid)`

输出两样东西：

- `FbRelationInfo`
- `TupleDesc`

这一步结束后，入口已经知道：

- 目标 relation 支不支持
- 走 `keyed` 还是 `bag`
- 有没有 TOAST relation
- 后续输出行的 tuple descriptor 是什么

## 六、入口如何进入 WAL / replay 主链

关键函数：

- `fb_build_flashback_reverse_ops()`

这个函数是当前主链收口点，内部固定顺序是：

1. `fb_wal_prepare_scan_context()`
2. `fb_wal_build_record_index()`
3. 检查 `index.unsafe`
4. `fb_reverse_source_create()`
5. `fb_replay_build_reverse_source()`
6. `fb_reverse_source_finish()`

注意这里和早期文档的差异：

- 当前入口不再显式先构建完整 `ForwardOp array`
- 也不再先产出结果表
- 而是直接把 replay 结果收敛进 `FbReverseOpSource`

## 七、unsafe window 在入口层怎么拦

`fb_wal_build_record_index()` 返回后，如果：

```c
index.unsafe == true
```

入口层会调用 `fb_build_unsafe_detail()` 生成 detail，再直接报错。

detail 当前会包含：

- `scope`
- `operation`
- `target`
- `toast`
- `xid`
- `commit_ts`
- `lsn`

这使得入口层虽然不负责判断 unsafe，但负责把错误组织成最终对用户可见的形式。

## 八、`FbFlashbackQueryState` 保存什么

`src/fb_entry.c` 顶部定义了：

```c
typedef struct FbFlashbackQueryState
```

当前保存：

- `FbRelationInfo info`
- `TupleDesc tupdesc`
- `FbSpoolSession *spool`
- `FbReverseOpSource *reverse`
- `FbApplyContext *apply`
- `ExprContext *econtext`
- `bool cleanup_registered`

这基本就是“一次 flashback SRF 执行需要跨调用保留的状态”。

## 九、为什么入口要自己管理 cleanup callback

当前 `pg_flashback()` 是长生命周期 SRF。
如果中途异常、提前退出、执行器中断，不做 cleanup 就会留下：

- 未关闭的 apply relation scan
- 未销毁的 reverse source
- 未删除的 spool 目录
- 未结束的 progress context

因此入口层显式做了：

- `RegisterExprContextCallback(...)`
- `fb_flashback_cleanup_callback(...)`
- `fb_unregister_flashback_cleanup(...)`

这是当前实现稳定性的关键点之一。

## 十、value-per-call 分支怎么跑

当执行器没有给 materialize 分支时，入口走标准 SRF：

### 首次调用

在 `SRF_IS_FIRSTCALL()` 分支里：

1. 校验 `ReturnSetInfo`
2. 要求 `SFRM_ValuePerCall`
3. 初始化 `FuncCallContext`
4. 创建 `FbFlashbackQueryState`
5. `fb_progress_begin()`
6. 注册 cleanup callback
7. 调 `fb_prepare_flashback_query()`

而 `fb_prepare_flashback_query()` 又会做：

1. `fb_spool_session_create()`
2. `fb_validate_flashback_target()`
3. `fb_build_flashback_reverse_ops()`
4. `fb_apply_begin()`
5. 释放 reverse source 和 spool session 的构建期资源

这里有个容易看漏的点：

- reverse source 在 `fb_apply_begin()` 之后就可以销毁
- 因为 keyed / bag apply 初始化已经把需要的 reverse-op 消费进自己的 mode state

### 每次取一行

在 `SRF_PERCALL_SETUP()` 后：

- 调 `fb_apply_next()`
- 有结果就 `SRF_RETURN_NEXT`
- 没结果就 `fb_apply_end()` 并 `SRF_RETURN_DONE`

## 十一、为什么现在只保留 value-per-call

当前实现已经删除内部 materialized SRF 分支。

原因是 live case 已确认：对大结果集，内部 `tuplestore` 会把 temp I/O 转移到 PostgreSQL `base/pgsql_tmp/*`，而这部分并不受 `pg_flashback.memory_limit` 控制。

因此当前入口层统一只保留：

1. `fb_progress_begin()`
2. `fb_prepare_flashback_query()`
3. `SRF_PERCALL_SETUP()`
4. `fb_apply_next()`
5. `SRF_RETURN_NEXT / SRF_RETURN_DONE`

但这里要注意一件事：

- “函数内部只走 `ValuePerCall`”
- 不等于 “`FROM pg_flashback(...)` 一定不会被 PostgreSQL 物化”

真正解决 `FROM` 子句 temp spill 的，是后面新增的 `CustomScan` 路径，而不是单靠 `ValuePerCall`。

## 十二、为什么新增了 CustomScan

root cause 已经确认过：

1. 扩展内部即使只返回 `ValuePerCall`
2. PostgreSQL 默认 `FunctionScan` 仍会调用 `ExecMakeTableFunctionResult()`
3. table SRF 的完整输出仍会先进 `tuplestore`
4. 结果大时继续写 `base/pgsql_tmp/*`

所以当前新增了：

- `fb_pg_flashback_support(internal)`：
  - 绑定到 `pg_flashback` 的 `SUPPORT` 子句
  - 让新会话在规划阶段就加载 `pg_flashback.so`
- `fb_custom_scan`：
  - 注册 `set_rel_pathlist_hook`
  - 识别 `RTE_FUNCTION`
  - 输出 `CustomPath / CustomScan`
  - executor 中直接复用 `fb_flashback_query_begin/next/finish`

这样 `count(*) FROM pg_flashback(...)` 就不再经过默认 `FunctionScan -> tuplestore`。

## 十三、apply 与入口的边界在哪里

入口层只负责：

- 建上下文
- 处理错误
- 驱动 SRF

真正恢复结果集的逻辑已经下沉到 apply 层：

- `fb_apply_begin()`
- `fb_apply_next()`
- `fb_apply_end()`

入口层在 `fb_build_flashback_reverse_ops()` 之后，就不再直接碰 WAL 和 replay。

## 十四、为什么入口层要特别处理 tuple -> Datum

当前 apply 有两种 emit 形式：

- `TupleTableSlot`
- `HeapTuple`

在 value-per-call 路径下，统一由 `fb_apply_emit_as_datum()` 处理：

- slot 用 `ExecFetchSlotHeapTupleDatum()`
- heap tuple 用 `heap_copy_tuple_as_datum()`

这里的 `heap_copy_tuple_as_datum()` 是当前实现里非常关键的一步。
早期如果直接把普通 heap tuple 当复合 Datum 返回，会崩。

## 十四、进度显示如何从入口层起步

入口层会调用：

- `fb_progress_begin()`
- `fb_progress_finish()`
- 异常时 `fb_progress_abort()`

并在 gate 前进入 stage 1。
后续 WAL / replay / apply 模块各自更新对应阶段。

所以进度系统的生命周期归入口层管，具体百分比归下游模块管。

## 十五、`fb_export_undo()` 当前处于什么状态

入口骨架已经在 `src/fb_entry.c` 里：

1. 校验时间
2. 校验 archive
3. 校验 relation
4. 建 spool
5. 准备 scan context
6. 建 record index
7. 检查 unsafe

然后直接：

```text
fb_raise_not_implemented("fb_export_undo page replay + reverse op stream")
```

也就是说：

- 架子有了
- 公开安装面没有
- 真正的 replay -> reverse-op -> undo render 还没接上

## 十六、读入口代码时最短路径

如果你只想最快抓住主链，按这个顺序读就够：

1. `pg_flashback()`
2. `fb_prepare_flashback_query()`
3. `fb_validate_flashback_target()`
4. `fb_build_flashback_reverse_ops()`
5. `fb_apply_begin()`
6. `fb_apply_next()`

## 十七、最短心智模型

当前入口层本质上只做三件事：

```text
把 SQL 调用翻译成 relation + target_ts
把中间主链驱动到 apply ready
把 apply 结果安全地交给 SRF 发射
```
