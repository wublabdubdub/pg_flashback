# 核心入口源码导读

本文只讲当前入口：

```sql
SELECT *
FROM pg_flashback(
  NULL::public.t1,
  '2026-03-23 08:09:30+08'
);
```

不再覆盖已经删除的“结果表名 + 结果表写入”老入口。

## 入口做了什么

`src/fb_entry.c` 的 `pg_flashback(PG_FUNCTION_ARGS)` 当前负责 5 件事：

1. 解析目标 relation row type
2. 解析目标时间
3. 进入 gate，并构建 `ReverseOp`
4. 初始化流式 apply 上下文
5. 按 SRF value-per-call 逐行返回

## 第一步：从 `NULL::schema.table` 解析 relation

关键函数：

- `fb_resolve_target_type_relid()`

作用：

- 读取第一个参数在表达式层已经解析出的类型 Oid
- 用 `typeidTypeRelid()` 反推出目标表 `relid`

这就是为什么调用形态固定为：

```sql
NULL::public.t1
```

而不是：

- 字符串表名
- `regclass`
- `AS t(...)`

## 第二步：解析目标时间

关键函数：

- `fb_parse_target_ts_text()`

当前仍接受 `text`，由 PostgreSQL 内置的 `timestamptz_in` 完成解析。

## 第三步：统一 gate

关键函数：

- `fb_validate_flashback_target()`

它串起：

- `fb_require_target_ts_not_future()`
- `fb_require_archive_dir()`
- `fb_catalog_load_relation_info()`
- `fb_relation_tupledesc()`

完成后得到：

- 目标表是否支持
- `keyed` 还是 `bag`
- 查询需要的 `TupleDesc`

## 第四步：构建 `ReverseOp`

关键函数：

- `fb_build_flashback_reverse_ops()`

内部顺序固定为：

1. `fb_wal_prepare_scan_context()`
2. `fb_wal_build_record_index()`
3. `fb_build_forward_ops()`
4. `fb_build_reverse_ops()`

这一步完成后，入口层不再直接碰 WAL。

## 第五步：启动流式 apply

关键函数：

- `fb_apply_begin()`
- `fb_apply_next()`
- `fb_apply_end()`

当前实现不是：

- 先算完整张历史表
- 再统一输出

而是：

- 扫当前表
- 未命中变化集的当前行直接返回
- 命中变化集的行按 reverse-op 计算历史行
- 扫描结束后补发 residual 历史行

## 为什么要用 SRF

当前 `pg_flashback()` 是 value-per-call SRF。

这样可以：

- 不落最终结果表
- 不需要 `tuplestore`
- 首行可以更早返回
- 保持结果集接口

为避免直接把“普通 heap tuple”错误当成复合 Datum 返回，当前入口在返回前统一调用：

- `heap_copy_tuple_as_datum()`

这一步是当前实现稳定性的关键点。

## apply 阶段的当前分流

### keyed

实现文件：

- `src/fb_apply_keyed.c`

当前策略：

- 建立 `changed key -> action list`
- 扫当前表时只检查命中的 key
- residual 阶段只补发历史上存在但当前不存在的 key

### bag

实现文件：

- `src/fb_apply_bag.c`

当前策略：

- 建立 `row_identity -> delta_count`
- 当前扫描时吞掉需要抵消的当前行
- residual 阶段补发剩余正 delta

## 进度上报

`src/fb_progress.c` 当前固定输出 9 段：

1. 前置校验
2. 准备 WAL 扫描上下文
3. 扫描 WAL 并建立索引
4. replay discover
5. replay warm
6. replay final 并生成 `ForwardOp`
7. 构建 `ReverseOp`
8. 应用 reverse-op
9. 发射 residual 历史结果

注意：

- stage `9` 已不再代表“结果表物化”
- 而是 residual 行发射

## 读入口代码时的最短心智模型

可以只记这条线：

```text
pg_flashback(anyelement, text)
  -> gate
  -> WAL index
  -> page replay
  -> ForwardOp
  -> ReverseOp
  -> streaming apply
  -> SRF rows
```

如果你在入口层看到任何“创建结果表 / 返回表名 / 并行写表”的描述，那已经不是当前代码。
