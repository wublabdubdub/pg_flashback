# 核心入口源码导读

这份文档只讲一条线：

- 用户把 3 个 `text` 参数传给 `pg_flashback(...)`
- 代码怎样把它们解析成内部对象
- 中间怎样经过 relation gate、WAL 索引、page replay、reverse op、apply
- 最后怎样把结果写进结果表并返回表名

和上一版不同，这一版重点不是“告诉你有哪些函数”，而是：

- 把真正关键的代码直接贴出来
- 每一行都加注释
- 在代码块前后补更直白的解释

如果你读完还想继续深挖，可以再看：

- `docs/architecture/源码级维护手册.md`
- `docs/architecture/调试与验证手册.md`

## 先给你一张总流程图

当前主链可以先记成这 9 步：

1. 用户执行 `pg_flashback(result_name, source_table, target_ts_text)`
2. 入口层把 3 个 `text` 参数解析成：
   - 结果表名字符串
   - 源表 `Oid`
   - 目标时间点 `TimestampTz`
3. relation gate 检查目标表是否支持，并决定它是 `keyed` 还是 `bag`
4. WAL 层建立扫描上下文，解析这次查询要读哪些 segment
5. WAL 层扫描 `(target_ts, query_now_ts]`，构建 `FbWalRecordIndex`
6. replay 层用 `checkpoint + FPI + block redo` 重建目标 relation 的历史页状态
7. replay 层从页变化提取 `ForwardOp`
8. reverse 层把 `ForwardOp` 反转成 `ReverseOp`
9. apply 层把 `ReverseOp` 作用到“当前表”，算出历史结果，并按串行或并行路径写入结果表

后面你看代码时，就沿着这 9 步走，不要跳。

## 1. 参数是怎么进来的

先看 3 个最基础的辅助函数。它们解决的是：

- 第 1 个参数结果表名怎么检查
- 第 2 个参数源表名怎么转成 `Oid`
- 第 3 个参数时间文本怎么转成 `TimestampTz`

来源：`src/fb_entry.c`

```c
static Oid                                                     // 返回值是 Oid，也就是 PostgreSQL 内部对象编号。
fb_resolve_relation_name(text *name_text,                      // 输入是用户传入的 text 表名。
						 LOCKMODE lockmode,                    // 调用者决定加什么锁。
						 bool missing_ok)                     // 调用者决定找不到时是否报错。
{                                                              // 函数体开始。
	List	   *names;                                          // 用来保存分解后的 schema/table 名称列表。
	RangeVar   *rv;                                             // PostgreSQL 里表示“关系名引用”的结构。

	names = textToQualifiedNameList(name_text);                 // 把 text 拆成 qualified name list，例如 public.t1 -> ["public","t1"]。
	rv = makeRangeVarFromNameList(names);                       // 再把名字列表包装成 RangeVar。
	return RangeVarGetRelid(rv, lockmode, missing_ok);         // 最后让 PostgreSQL 解析出真正的 relid。
}                                                              // 函数结束。

static char *                                                  // 返回规范化后的结果表名字符串。
fb_require_result_name(text *name_text)                        // 输入是用户给的第一个参数。
{                                                              // 函数体开始。
	List	   *names = textToQualifiedNameList(name_text);     // 先把 text 拆成名字列表。

	if (list_length(names) != 1)                               // 如果名字不是单段，例如 public.fb1，就不允许。
		ereport(ERROR,                                          // 直接抛错。
				(errcode(ERRCODE_INVALID_NAME),                  // 错误码：名字非法。
				 errmsg("flashback result relation name must be unqualified"))); // 明确要求结果表名不能带 schema。

	return pstrdup(strVal(linitial(names)));                   // 取出唯一那一段名字，复制成 C 字符串返回。
}                                                              // 函数结束。

static TimestampTz                                             // 返回 PostgreSQL 内部时间类型。
fb_parse_target_ts_text(text *target_ts_text)                  // 输入是用户传进来的第 3 个 text 参数。
{                                                              // 函数体开始。
	char	   *target_ts_cstr = text_to_cstring(target_ts_text); // 先把 text 转成普通 C 字符串。

	return DatumGetTimestampTz(DirectFunctionCall3(timestamptz_in, // 直接调用 PostgreSQL 自己的 timestamptz 输入函数。
												   CStringGetDatum(target_ts_cstr), // 第 1 个参数：时间字符串本身。
												   ObjectIdGetDatum(InvalidOid),    // 第 2 个参数：类型 Oid，不指定。
												   Int32GetDatum(-1)));             // 第 3 个参数：typmod，不指定。
}                                                              // 函数结束。
```

先把这 3 个函数理解透：

- 第一个负责“名字解析”
- 第二个负责“结果表名约束”
- 第三个负责“时间解析”

它们都没有任何 flashback 逻辑，它们只是把用户输入变成 PostgreSQL 内部可用的对象。

## 2. 用户入口 `pg_flashback()` 到底做了什么

这段是你必须反复看的第一段主代码。

来源：`src/fb_entry.c`

```c
Datum
pg_flashback(PG_FUNCTION_ARGS)
{
	text	   *result_name_text = PG_GETARG_TEXT_PP(0);      // 第 1 个参数：结果表名。
	text	   *source_name_text = PG_GETARG_TEXT_PP(1);      // 第 2 个参数：源表名。
	text	   *target_ts_text = PG_GETARG_TEXT_PP(2);        // 第 3 个参数：目标时间点文本。
	char	   *result_name;                                  // 规范化后的结果表名。
	Oid			source_relid;                                  // 源表 Oid。
	TimestampTz target_ts;                                   // 解析后的目标时间。
	FbRelationInfo info;                                     // relation gate 结果。
	TupleDesc	tupdesc;                                       // 源表结构。
	char	   *create_sql;                                   // CREATE UNLOGGED TABLE 语句。
	FbReverseOpStream reverse;                               // 统一主链产出的 ReverseOp。
	bool		use_parallel = false;                           // 是否走并行结果路径。
	uint64		result_count;                                  // 最终写出的结果行数。

	result_name = fb_require_result_name(result_name_text);  // 结果表名必须未限定。
	source_relid = fb_resolve_relation_name(source_name_text,
											 AccessShareLock,
											 false);         // 解析源表 Oid。
	target_ts = fb_parse_target_ts_text(target_ts_text);      // 解析目标时间文本。
	fb_validate_flashback_target(source_relid, target_ts,
								   &info, &tupdesc);         // 做 gate，并拿到 tupledesc。
	create_sql = fb_build_create_table_sql(result_name,
											 tupdesc);        // 先准备建表 SQL。

	fb_build_flashback_reverse_ops(target_ts, &info,
									 tupdesc, &reverse);    // 统一完成 WAL / replay / reverse-op 主链。
	use_parallel = fb_parallel_apply_workers() > 0;           // 看是否显式开启并行 apply/write。

	if (use_parallel)                                         // 若开启并行：
		result_count = fb_parallel_apply_reverse_ops(result_name,
													 create_sql,
													 &info,
													 tupdesc,
													 &reverse); // 交给 bgworker 编排层。
	else                                                      // 否则走默认串行路径：
		result_count = fb_apply_reverse_ops(&info,
											 tupdesc,
											 &reverse,
											 &sink);          // 当前 backend 自己写结果表。

	PG_RETURN_TEXT_P(cstring_to_text(result_name));           // 统一返回结果表名。
}
```

这一段你要抓住两个关键事实：

### 事实 1：主链结果不是直接返回 `SETOF record`

它的返回值其实只是一个 `text`：

- 内容就是结果表名

真正的数据已经被插入到：

- 用户传入名称对应的 `UNLOGGED` 结果表

也就是：

```sql
SELECT pg_flashback('fb1', 't1', '...');
SELECT * FROM fb1;
```

这是两步式接口，不是一步式 SRF。

### 事实 2：真正的 flashback 逻辑先收敛到 `ReverseOp`

`pg_flashback()` 做的是：

- 参数处理
- relation gate
- 调 `fb_build_flashback_reverse_ops()` 完成：
  - WAL 扫描
  - page replay
  - `ForwardOp -> ReverseOp`
- 最后再根据 `pg_flashback.parallel_apply_workers` 选择：
  - 串行 apply/write
  - 并行 apply/write

它并不直接碰 WAL，也不直接做 replay。

## 3. 核心主链：`fb_build_flashback_reverse_ops()`

来源：`src/fb_entry.c`

```c
static void                                                   // 无返回值，结果写到 reverse。
fb_build_flashback_reverse_ops(TimestampTz target_ts,         // 输入：目标时间。
							   const FbRelationInfo *info,    // 输入：已完成 gate 的 relation info。
							   TupleDesc tupdesc,             // 输入：已复制好的表结构。
							   FbReverseOpStream *reverse)    // 输出：最终 ReverseOp 流。
{                                                             // 函数体开始。
	FbWalScanContext scan_ctx;                                 // WAL 扫描上下文。
	FbWalRecordIndex index;                                    // WAL 扫描后建立的 RecordRef 索引。
	FbForwardOpStream forward;                                 // page replay 提取出的正向操作流。

	fb_wal_prepare_scan_context(target_ts, &scan_ctx);         // 固定这次查询的 WAL 扫描上下文。
	fb_wal_build_record_index(info, &scan_ctx, &index);        // 扫描 WAL，建立 RecordRef 索引。
	if (index.unsafe)                                             // 如果窗口内出现不支持的 rewrite / truncate / storage change。
		ereport(ERROR,                                            // 直接报错，不做 best-effort。
				(errcode(ERRCODE_FEATURE_NOT_SUPPORTED),          // 错误码：特性不支持。
				 errmsg("fb does not support WAL windows containing %s operations", // 明确给出 unsafe 原因。
						fb_wal_unsafe_reason_name(index.unsafe_reason))));

	fb_build_forward_ops(info, &index, tupdesc, &forward);       // 用 replay 内核从 RecordRef 生成 ForwardOp。
	fb_build_reverse_ops(&forward, reverse);                     // 把 ForwardOp 反转成 ReverseOp。
}                                                                // 函数结束。
```

这一段就是整个项目的最短摘要：

- 前 5 行是真正的前置 gate 和 WAL 阶段
- 最后 2 行是 page replay -> `ForwardOp` -> `ReverseOp`

注意：这里已经不直接做 apply。  
当前入口主链先把“历史事实”收敛成 `ReverseOp`，然后再决定最后怎么把结果算出来、写出去。

以后你调 bug 时，可以先问自己：

- 是前半段的输入事实错了？
- 还是后半段的 replay / apply 错了？

## 4. relation gate 和模式选择是怎么决定的

如果你不理解 `info.mode` 从哪来，后面的 `keyed` / `bag` 代码就会看不明白。

来源：`src/fb_catalog.c`

```c
static FbApplyMode                                              // 返回 keyed 或 bag。
fb_catalog_choose_mode(Relation rel, FbRelationInfo *info)      // 输入是已经打开的目标 relation 和待填充的 info。
{                                                               // 函数体开始。
	if (fb_catalog_find_stable_unique_key(rel, info))            // 如果找到了稳定唯一键。
		return FB_APPLY_KEYED;                                   // 就走 keyed 模式。

	return FB_APPLY_BAG;                                        // 否则回退到 bag 模式。
}                                                               // 函数结束。

void                                                             // 无返回值，作用是把 info 填满。
fb_catalog_load_relation_info(Oid relid, FbRelationInfo *info)   // 输入是源表 Oid 和待输出的 relation info。
{                                                                // 函数体开始。
	Relation rel;                                                // 目标表 relation。
	Relation toast_rel = NULL;                                   // toast relation，默认没有。
	FbApplyMode mode;                                            // 记录最终选择的模式。

	if (info == NULL)                                            // 防御性检查。
		elog(ERROR, "FbRelationInfo must not be NULL");           // info 不能为空。

	MemSet(info, 0, sizeof(*info));                              // 先把输出结构清零。

	rel = relation_open(relid, AccessShareLock);                 // 打开目标表。
	fb_catalog_require_supported_relation(rel);                  // 检查是不是普通持久化 heap 表。

	mode = fb_catalog_choose_mode(rel, info);                    // 选择 keyed 还是 bag。

	info->relid = relid;                                         // 记录主表 Oid。
	info->toast_relid = rel->rd_rel->reltoastrelid;              // 记录 toast 表 Oid。
	info->locator = rel->rd_locator;                             // 记录主表物理 locator。
	info->mode = mode;                                           // 写入最终模式。
	info->mode_name = (mode == FB_APPLY_KEYED) ? "keyed" : "bag"; // 写入模式名字符串。

	if (OidIsValid(info->toast_relid))                           // 如果这张表有 toast 子表。
	{                                                            // 分支开始。
		toast_rel = relation_open(info->toast_relid, AccessShareLock); // 打开 toast relation。
		info->toast_locator = toast_rel->rd_locator;             // 记录 toast 的物理 locator。
		info->has_toast_locator = true;                          // 标记：确实存在 toast locator。
		relation_close(toast_rel, AccessShareLock);              // 关闭 toast relation。
	}                                                            // 分支结束。

	relation_close(rel, AccessShareLock);                        // 关闭主表。
}                                                                // 函数结束。
```

读完这段，你就应该知道：

- `info.mode` 不是用户传的
- `info.mode` 是代码自己根据目标表元信息推导出来的
- `info` 里还顺带带上了 toast relation 的定位信息

所以 `FbRelationInfo` 是整个主链最早被填充、并且会一路往下传的核心对象。

## 5. WAL 扫描上下文是怎么建的

来源：`src/fb_wal.c`

```c
void                                                             // 无返回值，直接写入 ctx。
fb_wal_prepare_scan_context(TimestampTz target_ts, FbWalScanContext *ctx) // 输入是目标时间点和待填充的扫描上下文。
{                                                                // 函数体开始。
	if (ctx == NULL)                                              // 防御性检查。
		elog(ERROR, "FbWalScanContext must not be NULL");         // ctx 不能为空。

	MemSet(ctx, 0, sizeof(*ctx));                                // 先清空整个上下文。
	ctx->target_ts = target_ts;                                   // 固定用户给的目标时间点。
	ctx->query_now_ts = GetCurrentTimestamp();                    // 固定“本次查询现在时刻”。
	ctx->parallel_segment_scan_enabled = fb_parallel_segment_scan_enabled(); // 读当前 GUC，看是否启用 segment prefilter。
	ctx->current_segment_may_hit = true;                          // 默认认为当前 segment 有可能命中。

	fb_collect_archive_segments(ctx);                             // 真正去解析 archive_dest / pg_wal / recovered_wal。
}                                                                // 函数结束。
```

这一步虽然短，但作用非常大：

- 它把“用户传进来的一个时间点”转成“这次查询完整的 WAL 运行时上下文”
- 它也固定了 `query_now_ts`

为什么 `query_now_ts` 要固定？

- 因为 flashback 的窗口其实不是只靠 `target_ts`
- 真正窗口是 `(target_ts, query_now_ts]`
- 如果查询过程中 `now()` 漂移，就会导致同一轮查询边界不稳定

## 6. WAL 事实层：`fb_wal_build_record_index()`

这一步非常重要，因为它生成的 `FbWalRecordIndex` 是 replay 的直接输入。

当前 `pg_flashback()` 进入这一层前，已经会通过 `fb_progress` 向 `psql` 输出阶段进度；这里对应固定 `9` 段模型里的第 `3` 段，并按实际访问的 WAL segment 推进百分比。

来源：`src/fb_wal.c`

```c
void                                                             // 无返回值，直接填充 index。
fb_wal_build_record_index(const FbRelationInfo *info,            // 输入：relation 元信息。
						  FbWalScanContext *ctx,                // 输入：WAL 扫描上下文。
						  FbWalRecordIndex *index)             // 输出：RecordRef 索引总表。
{                                                                // 函数体开始。
	FbWalIndexBuildState state;                                  // visitor 运行时状态。
	FbWalVisitWindow *windows = NULL;                            // 如果启用 prefilter，会生成命中窗口数组。
	uint32		window_count = 0;                                  // 命中窗口数量。
	uint32 i;                                                    // 循环变量。

	if (info == NULL)                                            // 防御性检查。
		elog(ERROR, "FbRelationInfo must not be NULL");           // relation info 不能为空。
	if (ctx == NULL)                                             // 防御性检查。
		elog(ERROR, "FbWalScanContext must not be NULL");        // scan context 不能为空。
	if (index == NULL)                                           // 防御性检查。
		elog(ERROR, "FbWalRecordIndex must not be NULL");        // index 不能为空。

	MemSet(index, 0, sizeof(*index));                            // 先把整个 index 清零。
	index->memory_limit_bytes = fb_get_memory_limit_bytes();      // 记录本次查询的 memory limit。
	index->xid_statuses = fb_create_xid_status_hash();            // 创建 xid -> commit/abort 状态表。

	MemSet(&state, 0, sizeof(state));                            // 初始化 visitor state。
	state.info = info;                                           // 带上 relation info。
	state.ctx = ctx;                                             // 带上 scan context。
	state.index = index;                                         // 带上输出 index。
	state.touched_xids = fb_create_touched_xid_hash();           // 创建 touched xid 集合。
	state.unsafe_xids = fb_create_unsafe_xid_hash();             // 创建 unsafe xid 集合。

	fb_prepare_segment_prefilter(info, ctx);                     // 先准备 segment prefilter。
	window_count = fb_build_prefilter_visit_windows(ctx, &windows); // 根据 hit map 构造实际扫描窗口。
	if (window_count == 0)                                       // 如果没有缩窗。
		fb_wal_visit_records(ctx, fb_index_record_visitor, &state); // 就完整扫描全部需要的 WAL。
	else                                                         // 否则进入命中窗口扫描路径。
	{                                                            // 分支开始。
		for (i = 0; i < window_count; i++)                       // 逐个扫描命中窗口。
		{                                                        // 循环开始。
			fb_wal_visit_window(ctx, &windows[i], fb_index_record_visitor, &state); // 对单个窗口做有序 record 访问。
			if (ctx->unsafe)                                      // 一旦发现 unsafe。
				break;                                           // 就提前停掉。
		}                                                        // 循环结束。
		pfree(windows);                                           // 释放窗口数组。
	}                                                            // 分支结束。

	if (!ctx->anchor_found)                                      // 如果没找到 target_ts 之前最近 checkpoint。
		ereport(ERROR,                                           // 直接报错。
				(errcode(ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE), // 错误码：前提条件不满足。
				 errmsg("WAL not complete: no checkpoint before target timestamp"))); // 明确指出缺 checkpoint anchor。

	index->anchor_found = ctx->anchor_found;                     // 回填 anchor 是否找到。
	index->anchor_checkpoint_lsn = ctx->anchor_checkpoint_lsn;   // 回填 checkpoint record 的 LSN。
	index->anchor_redo_lsn = ctx->anchor_redo_lsn;               // 回填 checkpoint redo 起点。
	index->anchor_time = ctx->anchor_time;                       // 回填 anchor 时间。
	index->unsafe = ctx->unsafe;                                 // 回填 unsafe 标记。
	index->unsafe_reason = ctx->unsafe_reason;                   // 回填 unsafe 原因。
```

接下来它会给每条 `FbRecordRef` 补事务状态：

```c
	for (i = 0; i < index->record_count; i++)                    // 遍历所有已保留的 RecordRef。
	{                                                            // 循环体开始。
		FbRecordRef *record = &index->records[i];                // 取当前 record。
		FbXidStatusEntry *status_entry;                          // 用来查这个 xid 的 commit/abort 结果。

		if (record->lsn < index->anchor_redo_lsn)                // anchor 之前的 record 不参与正式 replay。
			continue;                                            // 跳过。

		status_entry = (FbXidStatusEntry *) hash_search(index->xid_statuses, // 在 xid 状态 hash 里查当前 xid。
														&record->xid,       // key 就是当前 record 的 xid。
														HASH_FIND, NULL);   // 只查，不插入。
		if (status_entry != NULL)                                // 如果查到了事务最终状态。
		{                                                        // 分支开始。
			record->commit_ts = status_entry->commit_ts;         // 回填 commit/abort 时间。
			record->commit_lsn = status_entry->commit_lsn;       // 回填 commit record 的 LSN。
			record->aborted = (status_entry->status == FB_WAL_XID_ABORTED); // 标记这条 record 是否属于已回滚事务。
			record->committed_after_target =                    // 标记：它是否属于 target_ts 之后提交的事务。
				(status_entry->status == FB_WAL_XID_COMMITTED && // 前提 1：事务已提交。
				 status_entry->commit_ts > ctx->target_ts &&    // 前提 2：提交时间晚于 target_ts。
				 status_entry->commit_ts <= ctx->query_now_ts); // 前提 3：提交时间不晚于本次查询时刻。
			record->committed_before_target =                   // 标记：它是否属于 target_ts 之前已提交事务。
				(status_entry->status == FB_WAL_XID_COMMITTED && // 前提 1：事务已提交。
				 status_entry->commit_ts <= ctx->target_ts);    // 前提 2：提交时间不晚于 target_ts。
		}                                                        // 分支结束。
```

你要把这一步理解成：

- 它并不直接计算最终结果
- 它在做“事实整理”

也就是把原始 WAL record 整理成：

- 哪些跟目标表有关
- 哪些事务已提交
- 哪些要回滚
- 哪些是 replay 必需记录

## 7. replay 总控：`fb_replay_execute_internal()`

这段代码是当前实现里最重要的一段之一。  
你只要把这段看懂，就能明白为什么这个项目现在一定要走“页级重放内核”。

来源：`src/fb_replay.c`

```c
static void                                                     // 无返回值，结果通过 result / stream 输出。
fb_replay_execute_internal(const FbRelationInfo *info,          // 输入：relation 元信息。
						   const FbWalRecordIndex *index,      // 输入：WAL 事实层索引。
						   TupleDesc tupdesc,                  // 输入：主表结构，只有生成 ForwardOp 时才需要。
						   FbReplayResult *result,            // 输出：replay 统计结果。
						   FbForwardOpStream *stream)         // 输出：ForwardOp 流；如果为 NULL 就只做 replay 不产出 forward op。
{                                                               // 函数体开始。
	FbReplayStore store;                                         // 正式 replay 使用的 BlockReplayStore。
	FbToastStore *toast_store = NULL;                            // 正式 replay 使用的 TOAST store。
	FbToastStore *discover_toast_store = NULL;                   // discover 阶段使用的临时 TOAST store。
	Relation toast_rel = NULL;                                   // toast relation。
	TupleDesc toast_tupdesc = NULL;                              // toast relation 的表结构。
	HTAB *backtrack_blocks;                                      // 缺页基线 block 集合。
	uint32 iteration;                                            // 共享回扫轮次计数。
	uint64 warm_tracked_bytes;                                   // warm 阶段产生的 tracked bytes。

	if (stream != NULL)                                          // 如果调用方需要 ForwardOp。
	{                                                            // 分支开始。
		MemSet(stream, 0, sizeof(*stream));                      // 先把 stream 清零。
		stream->tracked_bytes = index->tracked_bytes;            // 继承 index 阶段已经计费的内存。
		stream->memory_limit_bytes = index->memory_limit_bytes;  // 继承同一条 query 的 memory limit。
	}                                                            // 分支结束。

	if (info->has_toast_locator && OidIsValid(info->toast_relid)) // 如果目标表带 TOAST。
	{                                                            // 分支开始。
		toast_rel = relation_open(info->toast_relid, AccessShareLock); // 打开 toast relation。
		toast_tupdesc = CreateTupleDescCopy(toast_rel->rd_att);  // 复制 toast 表结构。
		relation_close(toast_rel, AccessShareLock);              // 关闭 toast relation。
	}                                                            // 分支结束。

	backtrack_blocks = fb_replay_create_missing_block_hash();     // 创建“缺页基线 block 集合”。

	for (iteration = 0; iteration < 8; iteration++)              // 最多做 8 轮共享补锚。
	{                                                            // 循环体开始。
		MemoryContext discover_ctx;                              // discover 阶段的临时内存上下文。
		MemoryContext oldctx;                                    // 保存旧上下文。
		FbReplayStore discover_store;                            // discover 阶段自己的 BlockReplayStore。
		FbReplayControl discover_control;                        // discover 阶段控制对象。
		FbReplayResult discover_result;                          // discover 阶段统计对象。
		HTAB *iteration_missing;                                 // 本轮 discover 新发现的缺页块集合。

		discover_ctx = AllocSetContextCreate(CurrentMemoryContext, // 创建 discover 专用 memory context。
											 "fb replay discover", // 上下文名字。
											 ALLOCSET_DEFAULT_SIZES); // 默认大小配置。
		oldctx = MemoryContextSwitchTo(discover_ctx);             // 切换到 discover context。
		MemSet(&discover_store, 0, sizeof(discover_store));      // 清零 discover store。
		discover_store.blocks = fb_replay_create_block_hash();   // 为 discover 阶段创建 block hash。
		if (toast_tupdesc != NULL)                               // 如果有 toast 表结构。
			discover_toast_store = fb_toast_store_create();       // 创建 discover 阶段的 toast store。
		fb_replay_init_result(index, &discover_result);          // 初始化 discover 阶段结果统计。
		MemoryContextSwitchTo(oldctx);                           // 切回原来的 memory context。

		fb_replay_warm_store(info, index, toast_tupdesc, discover_toast_store, // 先用已知补锚结果预热 discover store。
							 &discover_store, backtrack_blocks, &discover_result); // 让 discover 阶段拥有尽可能完整的页基线。

		iteration_missing = fb_replay_create_missing_block_hash(); // 创建“本轮新发现缺页块”集合。
		MemSet(&discover_control, 0, sizeof(discover_control));   // 清零 discover 控制对象。
		discover_control.phase = FB_REPLAY_PHASE_DISCOVER;        // 当前阶段设为 DISCOVER。
		discover_control.missing_blocks = iteration_missing;      // 把本轮缺页块集合挂进去。
		fb_replay_run_pass(info, index, NULL, toast_tupdesc, discover_toast_store, // 跑一遍 discover pass。
						   &discover_store, &discover_control, &discover_result,    // 输出只进入 discover store / result。
						   NULL, index->anchor_redo_lsn, InvalidXLogRecPtr);        // 此阶段不产出 ForwardOp。

		if (fb_replay_missing_block_count(iteration_missing) == 0) // 如果这一轮没发现新的缺页块。
		{                                                          // 分支开始。
			hash_destroy(iteration_missing);                       // 销毁本轮缺页块集合。
			if (discover_toast_store != NULL)                      // 如果创建过 discover toast store。
				fb_toast_store_destroy(discover_toast_store);      // 就销毁它。
			discover_toast_store = NULL;                           // 指针置空。
			MemoryContextDelete(discover_ctx);                     // 整个 discover 内存上下文删除。
			break;                                                 // 共享补锚阶段结束。
		}                                                          // 分支结束。

		fb_replay_resolve_missing_anchors(index, iteration_missing); // 对这一轮缺页块回看更早 RecordRef，找最近 anchor。
		fb_replay_raise_unresolved_missing_fpi(index, iteration_missing); // 如果连更早回看都找不到，就直接报 missing FPI。
		fb_replay_merge_missing_blocks(backtrack_blocks, iteration_missing); // 把本轮缺页块并入总 backtrack 集合。
		hash_destroy(iteration_missing);                           // 销毁本轮缺页块集合。
		if (discover_toast_store != NULL)                          // 如果有 discover toast store。
			fb_toast_store_destroy(discover_toast_store);          // 销毁它。
		discover_toast_store = NULL;                               // 指针置空。
		MemoryContextDelete(discover_ctx);                         // 删除 discover 上下文。

		if (iteration == 7)                                        // 如果已经做到第 8 轮还没收敛。
			ereport(ERROR,                                        // 直接报错。
					(errcode(ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE), // 前提条件不满足。
					 errmsg("too many shared backtracking rounds while resolving missing FPI"))); // 明确指出是共享补锚轮次过多。
	}                                                              // 循环体结束。

	MemSet(&store, 0, sizeof(store));                              // 共享补锚阶段结束后，初始化正式 replay store。
	store.blocks = fb_replay_create_block_hash();                  // 创建正式 BlockReplayStore。
	fb_replay_init_result(index, result);                          // 初始化正式 replay 结果统计。
	if (toast_tupdesc != NULL)                                     // 如果目标表有 toast。
		toast_store = fb_toast_store_create();                     // 创建正式 TOAST store。

	fb_replay_warm_store(info, index, toast_tupdesc, toast_store,  // 正式 replay 前，先把共享补锚结果预热进正式 store。
						 &store, backtrack_blocks, result);       // 这样 FINAL 阶段就有页基线了。
	warm_tracked_bytes = Max(index->tracked_bytes, result->tracked_bytes); // 记住 warm 后的 tracked bytes 峰值。
	fb_replay_init_result(index, result);                          // 重新初始化 result，准备 FINAL 阶段重新统计。
	result->tracked_bytes = warm_tracked_bytes;                    // 但保留 warm 阶段已经产生的 tracked bytes 峰值。

	{                                                              // 单独开一个小作用域。
		FbReplayControl final_control;                             // FINAL 阶段控制对象。

		MemSet(&final_control, 0, sizeof(final_control));          // 清零 final control。
		final_control.phase = FB_REPLAY_PHASE_FINAL;               // 当前阶段设为 FINAL。
		fb_replay_run_pass(info, index, tupdesc, toast_tupdesc, toast_store, // 用正式 store 跑 FINAL 阶段。
						   &store, &final_control, result, stream,           // 这一次真正产出 ForwardOp。
						   index->anchor_redo_lsn, InvalidXLogRecPtr);       // 从 anchor redo 开始放到窗口末尾。
	}                                                              // 小作用域结束。

	if (stream != NULL && tupdesc != NULL)                         // 如果调用方确实要 ForwardOp。
	{                                                              // 分支开始。
		fb_forward_stream_finalize(info, tupdesc, toast_store, result, stream); // finalize：处理 toast、identity 等收尾。
		stream->tracked_bytes = result->tracked_bytes;             // 回填最终 tracked bytes。
		stream->memory_limit_bytes = result->memory_limit_bytes;   // 回填最终 memory limit。
	}                                                              // 分支结束。

	if (toast_store != NULL)                                       // 如果创建了正式 toast store。
		fb_toast_store_destroy(toast_store);                       // 销毁它。
}                                                                  // 函数结束。
```

这一大段代码你必须记住 3 件事：

### 1. 不是一次 replay 完事

它分两大阶段：

- 共享补锚阶段
- 正式 FINAL replay 阶段

### 2. `missing FPI` 不是一看到就立刻报

它会先：

- 发现缺页块
- 共享回看更早 `RecordRef`
- 找最近可恢复 anchor

只有共享补锚也解决不了，才报真正的 `missing FPI`

### 3. `ForwardOp` 是 FINAL 阶段才产出的

discover / warm 阶段只是为了把页状态准备对，  
真正的逻辑行变化提取是在 FINAL 阶段做的。

## 8. `ForwardOp -> ReverseOp` 是怎么转的

来源：`src/fb_reverse_ops.c`

```c
void                                                            // 无返回值，直接填充 reverse stream。
fb_build_reverse_ops(const FbForwardOpStream *forward,          // 输入：ForwardOp 流。
					 FbReverseOpStream *reverse)               // 输出：ReverseOp 流。
{                                                               // 函数体开始。
	uint32 i;                                                   // 循环变量。

	MemSet(reverse, 0, sizeof(*reverse));                       // 先把输出流清零。
	reverse->tracked_bytes = forward->tracked_bytes;            // 继承 forward 阶段的 tracked bytes。
	reverse->memory_limit_bytes = forward->memory_limit_bytes;  // 继承同一条 query 的 memory limit。

	for (i = 0; i < forward->count; i++)                        // 遍历全部 ForwardOp。
	{                                                           // 循环体开始。
		const FbForwardOp *forward_op = &forward->ops[i];       // 取当前 forward op。
		FbReverseOp reverse_op;                                 // 准备构造对应的 reverse op。

		MemSet(&reverse_op, 0, sizeof(reverse_op));             // 先清零。
		reverse_op.xid = forward_op->xid;                       // 复制事务号。
		reverse_op.commit_ts = forward_op->commit_ts;           // 复制提交时间。
		reverse_op.commit_lsn = forward_op->commit_lsn;         // 复制提交 LSN。
		reverse_op.record_lsn = forward_op->record_lsn;         // 复制原 record LSN。

		switch (forward_op->type)                               // 根据正向操作类型选择反向操作。
		{                                                       // switch 开始。
			case FB_FORWARD_INSERT:                             // 正向 INSERT。
				reverse_op.type = FB_REVERSE_REMOVE;            // 反向就是 REMOVE。
				reverse_op.new_row = forward_op->new_row;       // 要删掉的是“插入后的新行”。
				break;                                          // 当前分支结束。
			case FB_FORWARD_DELETE:                             // 正向 DELETE。
				reverse_op.type = FB_REVERSE_ADD;               // 反向就是 ADD。
				reverse_op.old_row = forward_op->old_row;       // 要加回的是“删除前的旧行”。
				break;                                          // 当前分支结束。
			case FB_FORWARD_UPDATE:                             // 正向 UPDATE。
				reverse_op.type = FB_REVERSE_REPLACE;           // 反向就是 REPLACE。
				reverse_op.old_row = forward_op->old_row;       // 目标值是更新前旧行。
				reverse_op.new_row = forward_op->new_row;       // 当前存在的是更新后新行。
				break;                                          // 当前分支结束。
		}                                                       // switch 结束。

		fb_reverse_stream_append(reverse, &reverse_op);         // 把当前 reverse op 追加进输出流。
	}                                                           // 循环体结束。

	if (reverse->count > 1)                                     // 如果有多条 reverse op。
		qsort(reverse->ops, reverse->count, sizeof(FbReverseOp), // 就按比较函数排序。
			  fb_reverse_op_cmp);                               // 排序规则是 commit_lsn DESC, record_lsn DESC。
}                                                               // 函数结束。
```

这一步其实非常纯粹：

- INSERT 反成 REMOVE
- DELETE 反成 ADD
- UPDATE 反成 REPLACE

如果你确认 replay 提取出的 `ForwardOp` 已经正确，但最终结果还是不对，  
就要回来看这一层和 apply 层。

## 9. apply 层是怎样把历史结果算出来的

最终结果不是“从 WAL 直接输出”，而是：

- 先读当前表
- 再把 `ReverseOp` 一条条应用上去

先看 apply 总入口。

来源：`src/fb_apply_keyed.c`

```c
uint64                                                          // 返回最终输出行数。
fb_apply_reverse_ops(const FbRelationInfo *info,                // 输入：relation info，里面有 mode。
					  TupleDesc tupdesc,                        // 输入：表结构。
					  const FbReverseOpStream *stream,          // 输入：ReverseOp 流。
					  const FbResultSink *sink)                // 输出：统一结果 sink。
{                                                               // 函数体开始。
	fb_progress_enter_stage(FB_PROGRESS_STAGE_APPLY, NULL);     // 进入第 8 阶段。
	if (info->mode == FB_APPLY_KEYED)                           // 如果这张表是 keyed 模式。
		return fb_apply_keyed_mode(info, tupdesc, stream, sink); // 走 keyed 算法。
	else                                                        // 否则。
		return fb_apply_bag_mode(info, tupdesc, stream, sink);   // 走 bag 算法。
}                                                               // 函数结束。
```

再看 keyed 模式主函数。

```c
uint64                                                          // 返回最终输出行数。
fb_apply_keyed_mode(const FbRelationInfo *info,                 // 输入：relation info。
					 TupleDesc tupdesc,                         // 输入：表结构。
					 const FbReverseOpStream *stream,           // 输入：ReverseOp 流。
					 const FbResultSink *sink)                // 输出：结果 sink。
{                                                               // 函数体开始。
	FbKeyedLoadState state;                                     // keyed 模式工作集。
	uint32 i;                                                   // 循环变量。
	HASH_SEQ_STATUS seq;                                        // hash 遍历状态。
	FbKeyBucket *bucket;                                        // keyed hash bucket。

	MemSet(&state, 0, sizeof(state));                           // 初始化工作集。
	state.info = info;                                          // 记录 relation info。
	state.tupdesc = tupdesc;                                    // 记录表结构。
	state.rows = fb_apply_create_bucket_hash("fb keyed rows");  // 创建 key -> tuple 的 hash。
	state.tracked_bytes = stream->tracked_bytes;                // 继承已跟踪内存。
	state.memory_limit_bytes = stream->memory_limit_bytes;      // 继承 memory limit。

	fb_apply_scan_current_relation(info->relid, fb_keyed_load_current_tuple, &state); // 先扫描当前表，建立“当前时刻工作集”。

	for (i = 0; i < stream->count; i++)                         // 遍历 ReverseOp 流。
	{                                                           // 循环体开始。
		const FbReverseOp *op = &stream->ops[i];                // 取当前 reverse op。

		switch (op->type)                                       // 看它是什么类型。
		{                                                       // switch 开始。
			case FB_REVERSE_REMOVE:                             // REMOVE：表示这行在 target_ts 不该存在。
				if (op->new_row.key_identity != NULL)           // 只要知道 key。
					fb_keyed_remove_row(state.rows, op->new_row.key_identity); // 就从当前工作集中删掉它。
				break;                                          // 当前分支结束。
			case FB_REVERSE_ADD:                                // ADD：表示这行在 target_ts 应该存在。
				if (op->old_row.key_identity != NULL && op->old_row.tuple != NULL) // 必须同时有 key 和 tuple。
					fb_keyed_upsert_row(&state, op->old_row.key_identity,          // 用旧 key。
										op->old_row.tuple, false, false);          // 把旧行补回工作集。
				break;                                          // 当前分支结束。
			case FB_REVERSE_REPLACE:                            // REPLACE：表示把新值换回旧值。
				if (op->new_row.key_identity != NULL)           // 如果知道新 key。
					fb_keyed_remove_row(state.rows, op->new_row.key_identity); // 先删掉当前新行。
				if (op->old_row.key_identity != NULL && op->old_row.tuple != NULL) // 如果知道旧 key 和旧 tuple。
					fb_keyed_upsert_row(&state, op->old_row.key_identity,          // 再把旧行放回去。
										op->old_row.tuple, false, false);
				break;                                          // 当前分支结束。
		}                                                       // switch 结束。
	}                                                           // 循环体结束。

	total_rows = fb_keyed_count_rows(state.rows);               // 先数最终结果行数。
	fb_progress_enter_stage(FB_PROGRESS_STAGE_MATERIALIZE, NULL); // 进入第 9 阶段。
	hash_seq_init(&seq, state.rows);                            // 准备遍历最终 keyed 工作集。
	while ((bucket = (FbKeyBucket *) hash_seq_search(&seq)) != NULL) // 遍历所有 bucket。
	{                                                           // 循环体开始。
		FbKeyedRow *row;                                        // bucket 内链表节点。

		for (row = (FbKeyedRow *) bucket->head; row != NULL; row = row->next) // 遍历该 bucket 里的所有行。
		{                                                       // 输出循环体。
			fb_result_sink_put_tuple(sink, row->tuple, tupdesc); // 直接写入结果表 sink。
			emitted++;                                           // 统计已写行数。
			fb_progress_update_fraction(FB_PROGRESS_STAGE_MATERIALIZE, // 推进第 9 段百分比。
										 emitted,
										 total_rows,
										 NULL);
		}
	}                                                           // 循环体结束。
	return emitted;                                             // 返回最终输出行数。
}                                                               // 函数结束。
```

这个 keyed 算法要这样理解：

- 当前表是“现在时刻”的工作集
- `ReverseOp` 一条条把它往回拨
- 最后工作集的内容，就是 `target_ts` 时刻该看到的结果

上面这段代码展示的是默认串行路径。  
如果开启 `pg_flashback.parallel_apply_workers > 0`，算法本质没变，只是 stage `8/9` 不再由入口 backend 单线程完成，而是 leader 把 current tuples 和 primitive reverse-op items 分发给多个 worker，各 worker 在自己的分区里完成同一套 keyed / bag apply，然后并行写同一张 `UNLOGGED` 结果表。

## 10. 最终结果是怎么返回给用户的

前面讲完之后，你现在就能回头理解 `pg_flashback()` 的最后几行了。

最关键的是这段：

来源：`src/fb_entry.c`

```c
	create_sql = fb_build_create_table_sql(result_name, tupdesc);             // 先拼出 CREATE UNLOGGED TABLE SQL。
	fb_build_flashback_reverse_ops(target_ts, &info, tupdesc, &reverse);      // 先统一产出 ReverseOp。
	use_parallel = fb_parallel_apply_workers() > 0;                           // 看当前是否显式开启并行 apply/write。

	if (use_parallel)                                                         // 并行路径：
		result_count = fb_parallel_apply_reverse_ops(result_name,              // 交给 bgworker 编排层。
													 create_sql,
													 &info,
													 tupdesc,
													 &reverse);
	else                                                                      // 默认串行路径：
	{                                                                         // 分支开始。
		/* 入口 backend 自己创建结果表，并把 sink 绑到 table insert */         // 串行路径下仍由当前 backend 负责建表和写表。
		result_count = fb_apply_reverse_ops(&info, tupdesc, &reverse, &sink);  // apply 末尾直接写表。
	}                                                                         // 分支结束。

	PG_RETURN_TEXT_P(cstring_to_text(result_name));                           // 最后统一返回 result_name 给用户。
```

你现在应该能彻底理解这个返回语义：

- 真正的数据不是通过函数返回值传回 SQL 层
- 真正的数据已经提前写进结果表
- 函数返回值只是告诉你：
  - “我把结果放进了哪个表”

另外你还要记住一件现在很关键的语义差异：

- `parallel_apply_workers = 0` 时，结果表由当前 backend 创建并写入
- `parallel_apply_workers > 0` 时，结果表会由独立 helper worker 自主事务创建并提交
- 所以并行路径下，结果表生命周期不再跟随调用方事务回滚

也正因为如此，用户调用习惯才是：

```sql
SELECT pg_flashback('fb1', 't1', '2026-03-23 10:00:00+08');
SELECT * FROM fb1;
```

## 11. 再把整条主链压缩成一句人话

如果你以后忘了，就回忆这一句：

> `pg_flashback()` 先把 3 个 `text` 参数解析成结果表名、源表 `Oid` 和目标时间，再完成 relation gate、WAL 索引、page replay、`ForwardOp -> ReverseOp`，最后按串行或并行路径把最终结果写进 `UNLOGGED` 结果表，并把结果表名返回给用户。
