# 源码级维护手册

## 这份手册解决什么问题

这份文档不是产品介绍，而是给维护者建立“怎么读、怎么改、怎么判断改动影响面”的源码地图。

读完后，你应该能回答这些问题：

- `pg_flashback(text, text, text)` 从哪里开始，最后怎样把结果写进结果表
  - 以及为什么并行路径下结果表不再跟随调用方事务回滚
- 为什么当前实现不是“直接从 WAL 拼 old tuple”，而是 `checkpoint + FPI + block redo -> ForwardOp -> ReverseOp`
- `keyed` / `bag` 两种执行模式分别在什么阶段分流
- 哪些结构体是跨模块传递的主脉络
- 出错时应该先怀疑 WAL resolver、record index、page replay、TOAST 还是 apply
- 以后新增一个 WAL record / 新增一个约束 / 修一个 replay bug，代码应该改哪些点

## 项目真正的技术主线

当前代码已经拍板的内核，不要再按旧思路理解：

- 最终对用户暴露的是逻辑层 `reverse op stream`
- 但 `reverse op stream` 不是直接从 WAL 行像 decode 出来的
- 当前真实来源是：
  - `target_ts` 前最近 checkpoint
  - checkpoint 之后收集到的 FPI / INIT_PAGE / block data / main data
  - 对目标 relation block 做 page redo
  - 从重放前/后页提取 `old_row` / `new_row`
  - 先得到 `ForwardOp`
  - 再转换成 `ReverseOp`
  - 再从“当前表”反推回 `target_ts`

这意味着：

- `fb_wal.c` 的职责不是直接产出用户结果，而是产出“足够 replay 的结构化事实”
- `fb_replay.c` 不是附属模块，而是当前正确性的中心
- `fb_apply_keyed.c` / `fb_apply_bag.c` 只负责“如何把 reverse stream 应用到当前结果集”，不负责理解 WAL

## 对外接口和安装面

当前安装 SQL `sql/pg_flashback--0.1.0.sql` 只安装 3 个函数：

- `fb_version()`
- `fb_check_relation(regclass)`
- `pg_flashback(text, text, text)`

未对外安装但代码中存在或预留的能力：

- `fb_export_undo()`：代码入口已存在，但仍是 `not implemented`
- `fb_flashback_materialize()` / `fb_internal_flashback()`：当前不安装
- 旧的 scan / replay / decode 调试 SQL 入口：已经从公开安装面收口

因此，当前用户侧真实入口只有：

```sql
SELECT pg_flashback('fb1', 'public.t1', '2026-03-23 08:09:30+08');
SELECT * FROM fb1;
```

## 贯穿主链的关键结构体

### `FbRelationInfo` in `include/fb_common.h`

这是 query 级 relation 静态元信息：

- `relid`
- `toast_relid`
- `locator`
- `toast_locator`
- `has_toast_locator`
- `mode`
- `mode_name`
- `key_natts`
- `key_attnums[]`

它回答三个问题：

- 目标表支不支持
- 是否有 TOAST 子表
- 结果应用时按 `keyed` 还是 `bag`

### `FbWalScanContext` in `include/fb_wal.h`

这是 WAL 扫描阶段的运行时上下文，保存：

- 时间窗：`target_ts` / `query_now_ts`
- segment 解析结果：timeline、首尾 segment、起止 LSN
- checkpoint anchor：`anchor_checkpoint_lsn` / `anchor_redo_lsn` / `anchor_time`
- 安全性：`unsafe` / `unsafe_reason`
- resolver 统计：archive/pg_wal/ckwal 使用情况
- prefilter 状态：hit map、命中 segment 数

注意：

- `FbWalScanContext` 不是 replay 结果
- 它描述的是“扫描这次 query 需要怎样的 WAL 访问上下文”

### `FbRecordBlockRef`

这是单条 WAL record 内单个 block 的抽象：

- block 属于哪个 relation / fork / blkno
- 是否主表块或 TOAST 块
- 是否有 block image
- 是否要求应用 image
- 是否有 block data

这层很关键，因为 replay 的输入不是“一个 heap_update record”，而是“一个 record + 其中关联的 block image / block data / main data”。

### `FbRecordRef`

这是 `fb_wal.c` 输出给 replay 的核心单位：

- `kind`
- `lsn` / `end_lsn`
- `xid`
- `info`
- `init_page`
- commit / abort 元信息
- `blocks[]`
- `main_data`

理解它最重要的一点：

- `FbRecordRef` 不是原始 `XLogReaderState`
- 它是“已经做过 relation 过滤、payload 复制、事务状态补充”的轻量 record 索引

### `FbWalRecordIndex`

这是整个查询窗口的索引总表，包含：

- anchor 信息
- unsafe 标记
- 所有 `FbRecordRef`
- `xid_statuses`
- 各种计数
- tracked memory bytes

`fb_replay.c` 的所有工作都建立在这个结构之上。

### `FbReplayStore` / `FbReplayBlockState`

这是一层 block hash：

- key = `RelFileLocator + ForkNumber + BlockNumber`
- value = 当前已 materialize 的 page 状态

只要一个 block 已被 FPI、INIT_PAGE 或更早回扫建立了页基线，它后续就会在这里被持续前向重放。

### `FbForwardOp` / `FbReverseOp`

`FbForwardOp`：

- 来源于 page replay 的逻辑变化
- 有 `type`
- 有 `record_lsn` / `commit_lsn`
- 有 `old_row` / `new_row`

`FbReverseOp`：

- `INSERT -> REMOVE`
- `DELETE -> ADD`
- `UPDATE -> REPLACE`

排序规则在 `fb_reverse_ops.c` 里固定为：

- `commit_lsn DESC`
- `record_lsn DESC`

### `FbToastStore`

这是 TOAST 历史值重建的内存/落盘存储：

- `live_chunks`
- `retired_chunks`
- `spill_file`

为什么需要 `retired_chunks`：

- 主表某次 `HOT_UPDATE` 之前，旧 TOAST chunk 可能已经被删掉
- 如果不把退休 chunk 留住，后续 old row finalize 时会丢历史值

## 从 SQL 到结果表的完整调用链

### 第 1 段：入口与参数处理

在 `src/fb_entry.c`：

1. `pg_flashback(PG_FUNCTION_ARGS)`
2. 校验目标结果表名必须是未限定名 `fb_require_result_name()`
3. 解析源表名 `fb_resolve_relation_name()`
4. 解析文本时间 `fb_parse_target_ts_text()`
5. 调主流程构建 `ReverseOp`

### 第 2 段：前置 gate 与 reverse-op 主链

`fb_build_flashback_reverse_ops()` 先做 4 件事：

1. `fb_require_target_ts_not_future()`
2. `fb_require_archive_dir()`
3. `fb_require_supported_target_relation()`
4. `fb_wal_prepare_scan_context()`

然后直接构建 `FbWalRecordIndex`：

```text
fb_wal_build_record_index()
```

如果扫描阶段已发现 `unsafe`，这里直接报错，不继续 replay。

### 第 3 段：WAL 索引层

`fb_wal_build_record_index()` 是第一层真正的核心：

1. 收集并解析 segment
2. 选择 timeline
3. `archive_dest + pg_wal + recovered_wal` 三路 resolve
4. 做 segment prefilter
5. 扫描 record
6. 找 `target_ts` 前最近 checkpoint
7. 记录 touched xid / xid status
8. 收集目标 relation 的 heap / heap2 / xlog FPI 记录
9. 标 unsafe window
10. 扫描结束后，把 commit / abort 元信息回填给每条 `FbRecordRef`

### 第 4 段：页级 replay

`fb_build_forward_ops()` 实际会调用：

```text
fb_replay_build_forward_ops()
  -> fb_replay_execute_internal()
```

`fb_replay_execute_internal()` 分三段：

1. `DISCOVER`
   - 从 checkpoint anchor 正常 replay
   - 如果某块还没有页基线，先记到 `missing_blocks`
2. `WARM`
   - 对缺页块共享回扫更早 record，找最近可用 `FPI/INIT_PAGE`
   - 预热 `BlockReplayStore`
3. `FINAL`
   - 从 anchor redo 正式重放
   - 对主表记录产出 `ForwardOp`
   - 对 TOAST 记录维护 `FbToastStore`

### 第 5 段：TOAST finalize

`fb_forward_stream_finalize()` 在 forward op 全部产生后执行：

- 若 tuple 仍含 external toast datum，则调 `fb_toast_rewrite_tuple()`
- 优先从历史 `FbToastStore` 拼 chunk
- chunk 不全时尝试 live TOAST fallback
- 最终必须得到 inline datum
- 如果 finalize 后还残留 external pointer，直接报错

### 第 6 段：反向流与结果应用

之后链路比较直接：

1. `fb_build_reverse_ops()`
2. `pg_flashback()` 根据 `pg_flashback.parallel_apply_workers` 选择结果路径：
   - 串行：
     - 入口 backend 自己创建 `UNLOGGED` 结果表
     - `fb_apply_reverse_ops()`
     - 按 relation mode 分发到 `fb_apply_keyed_mode()` / `fb_apply_bag_mode()`
   - 并行：
     - `fb_parallel_result_helper_main()` 自主事务创建并提交结果表
     - leader backend 扫当前表并拆分 / 派发 primitive reverse-op items
     - `fb_parallel_apply_worker_main()` 在各自分区里完成 keyed / bag apply
     - worker 直接并行写结果表

## 每个模块要抓住的实现要点

## `fb_entry.c`

你维护这个文件时要知道：

- 这里只做 orchestration，不做复杂业务逻辑
- 当前结果表默认是 `UNLOGGED heap`
- 当前结果输出不是 SRF，而是“返回表名 + 物化结果表”
- 当前入口已经不是单一路径：
  - `parallel_apply_workers = 0` 时走串行 sink
  - `parallel_apply_workers > 0` 时转到 `fb_parallel.c`

容易踩坑的点：

- 结果表名必须未限定，否则直接拒绝
- `pg_flashback()` 会先检查结果表是否已存在
- `fb_export_undo()` 现在还没有真正实现，别误以为安装面已经提供

## `fb_catalog.c`

`fb_catalog_load_relation_info()` 决定 relation 模式：

- 先排除 temp / unlogged / matview / partitioned parent / system catalog / 非 heap
- 再查 stable unique key
- 有稳定唯一键就 `keyed`
- 没有就 `bag`

稳定唯一键的定义很保守：

- primary key
- 或 immediate unique
- 不允许 predicate index
- 不允许 expression index

因此以后如果你放宽 keyed 规则，这个文件是第一入口。

## `fb_guc.c` / `fb_runtime.c` / `fb_ckwal.c` / `fb_progress.c`

这几个文件共同组成运行时环境与客户端可观测性。

### `fb_guc.c`

定义 7 个运行时配置：

- `pg_flashback.archive_dir`
- `pg_flashback.archive_dest`
- `pg_flashback.debug_pg_wal_dir`
- `pg_flashback.memory_limit_kb`
- `pg_flashback.parallel_segment_scan`
- `pg_flashback.show_progress`
- `pg_flashback.parallel_apply_workers`

`_PG_init()` 里还会直接调用 `fb_runtime_ensure_initialized()`，所以加载扩展时就会创建目录。

### `fb_runtime.c`

负责维护：

- `DataDir/pg_flashback/`
- `DataDir/pg_flashback/runtime/`
- `DataDir/pg_flashback/recovered_wal/`
- `DataDir/pg_flashback/meta/`

这些目录分别给：

- runtime spill
- recovered WAL
- prefilter meta cache

### `fb_ckwal.c`

当前不是“真正重建缺段”，而是：

- 从 archive/pg_wal/recovered_wal 找可信段
- 复制到 recovered_wal
- 或把错配 `pg_wal` 段按页头识别出真实 segno 后转正

它的能力边界非常重要：

- 只能复制 / 转换 / 回灌已有可用段
- 不能凭空恢复不存在的 WAL

### `fb_progress.c`

负责：

- 维护当前 backend 的 flashback progress context
- 统一输出 `NOTICE`
- 并在并行路径下汇总 stage `8/9` 的 worker 进度
- 固定 `9` 段阶段模型
- 对阶段 `3/4/5/6/7/8/9` 做 `20%` 粒度节流

注意它的边界：

- 只负责客户端进度可视化
- 不参与 WAL 选择、replay 正确性或 apply 结果计算
- 当前只服务 `pg_flashback()` 主入口

## `fb_wal.c`

这是最复杂的文件之一，可以按 5 个子系统理解。

### 1. payload arena

`FbWalRecordIndex` 里的 `main_data` / `block data` / `FPI image` 不再零散 `palloc`，而是通过 `fb_payload_arena_alloc()` 放进 arena。

原因：

- 减少热路径碎片化
- 统一 memory accounting

### 2. segment prefilter

关键函数：

- `fb_prepare_prefilter_patterns()`
- `fb_prepare_segment_prefilter()`
- `fb_build_prefilter_visit_windows()`

它做的不是“最终解析”，而是：

- 根据主表 locator 和 toast locator 的字节模式，先扫 segment 文件是否可能命中
- 命中结果写内存 cache，也写 `meta/` 目录
- 正式 `XLogReader` 只访问命中窗口附近的 segment

注意两个事实：

- prefilter 是保守优化，不允许破坏正确性
- 即使 prefilter miss，也仍会强制访问开头 segment，保证 checkpoint anchor 可被发现

### 3. source resolver

关键函数：

- `fb_collect_archive_segments()`
- `fb_validate_segment_identity()`
- `fb_try_ckwal_segment()`
- `fb_convert_mismatched_pg_wal_entry()`

当前规则：

- `archive_dest` 优先于 `pg_wal`
- `pg_wal` 只承接 archive 没覆盖的 tail
- `pg_wal` 文件名和页头不一致时视为 mismatch
- mismatch 段先转成 recovered_wal，再参与同一轮 resolver
- segment 缺口优先尝试 `fb_ckwal`
- 三路都拿不到，就报 `WAL not complete`

### 4. scan visitor

关键函数：

- `fb_scan_record_visitor()`
- `fb_note_checkpoint_record()`
- `fb_mark_record_xids_touched()`
- `fb_note_xact_record()`

这一层不产出 `RecordRef`，只做：

- 找 checkpoint anchor
- 识别 touched xid
- 识别 unsafe
- 统计 commit / abort

### 5. index visitor

关键函数：

- `fb_index_record_visitor()`
- `fb_copy_heap_record_ref()`
- `fb_copy_xlog_fpi_record_ref()`
- `fb_note_xact_status_for_touched()`

它在 scan visitor 的基础上多做一件事：

- 把目标 relation 真正需要 replay 的 record 复制成 `FbRecordRef`

当前已纳入的 record kind：

- heap:
  - `INSERT`
  - `DELETE`
  - `UPDATE`
  - `HOT_UPDATE`
  - `CONFIRM`
  - `LOCK`
  - `INPLACE`
- heap2:
  - `PRUNE_*`
  - `VISIBLE`
  - `MULTI_INSERT`
  - `LOCK_UPDATED`
- xlog:
  - `FPI`
  - `FPI_FOR_HINT`

`NEW_CID` 当前被显式识别，但不进入 replay。

### 6. unsafe 判定

当前直接报错的风险来源：

- `TRUNCATE`
- `REWRITE`
- `SMGR CREATE/TRUNCATE`
- target relation 的 `STANDBY LOCK`

注意：

- unsafe 不是“只要见到 record 就报”
- 只有和目标表相关，且最终在时间窗里形成有效事务边界时才转成 query 级失败

## `fb_replay.c`

这是当前 correctness 的核心文件。

### 1. BlockReplayStore

`fb_replay_get_block()` 会把 block materialize 到 hash。

页基线来源只有三种：

- record 自带 image
- `INIT_PAGE`
- 共享回扫找到的更早 image / init

如果这三种都没有，就会走 `missing FPI` 错误。

### 2. 共享回扫补锚

关键函数：

- `fb_replay_note_missing_block()`
- `fb_replay_resolve_missing_anchors()`
- `fb_replay_warm_store()`
- `fb_replay_raise_unresolved_missing_fpi()`

思路不是逐块无限回扫，而是：

1. 先跑一次 discover
2. 收集缺页块
3. 对缺页块共享回看更早 `RecordRef`
4. 找到每个 block 最近可用的 anchor record
5. 预热 store
6. 再跑正式 replay

这也是当前 batch B blocker 的主战场。

### 3. redo handlers

当前关键 redo 函数：

- `fb_replay_heap_insert()`
- `fb_replay_heap_delete()`
- `fb_replay_heap_update()`
- `fb_replay_heap_lock()`
- `fb_replay_heap2_prune()`
- `fb_replay_heap_confirm()`
- `fb_replay_heap_inplace()`
- `fb_replay_heap2_visible()`
- `fb_replay_heap2_lock_updated()`
- `fb_replay_xlog_fpi()`
- `fb_replay_heap2_multi_insert()`

维护时要记住：

- 主表块和 TOAST 块共用同一套 replay store
- 但只有主表 committed-after-target 的记录才会生成 `ForwardOp`
- TOAST 块主要驱动 `FbToastStore`

### 4. old/new row 提取

为什么 `DELETE/UPDATE` 能拿到 old row：

- 不是靠 WAL 自带 old tuple
- 是在 page redo 之前，从 replay 中的历史页把 offset 对应 tuple 拷出来

为什么 `INSERT/UPDATE` 能拿到 new row：

- redo 完之后再从页面拷 tuple

所以你以后如果看到结果不对，第一反应要判断：

- 是 page state 不对
- 还是 row image 抽取点不对

### 5. ForwardOp finalize

`fb_forward_stream_finalize()` 是 replay 之后、reverse 之前的最后一道关：

- 处理 external toast
- 生成 row/key identity
- 更新 memory accounting

这个阶段如果失败，通常不是 apply 层 bug，而是：

- TOAST 历史值不完整
- finalize 后仍残留 external pointer

## `fb_toast.c`

这份实现有 4 个关键点。

### 1. chunk store 分 live / retired

- `live_chunks`：当前在 replay 后仍存活的 chunk
- `retired_chunks`：历史中已删除，但 old row 仍可能需要引用的 chunk

### 2. retired chunk 会 spill 到 runtime 文件

`fb_toast_spill_chunk()` 会把 retired chunk 落到：

- `DataDir/pg_flashback/runtime/toast-retired-*.bin`

这样做是为了避免 retired history chunk 长驻内存。

### 3. tuple rewrite

`fb_toast_rewrite_tuple()` 会扫 tuple 的 varlena 列：

- 遇到 external on-disk datum
- 调 `fb_toast_reconstruct_datum()`
- 把 reconstructed datum inline 回 tuple

### 4. live fallback

如果历史 store 中 chunk 不全，代码会尝试：

- `fb_toast_fetch_live_datum()`

这是一个刻意保留的兜底：

- 某些未变更但仍 live 的 datum，历史 store 可能只覆盖到了部分 touched block
- 如果原 pointer 还能在 live TOAST relation 中完整解析，优先拿 live 值，而不是误报假缺失

这也是 TOAST 相关问题里最容易误判的一点。

## `fb_reverse_ops.c`

这个文件很简单，但它定义了最终查询语义：

- `INSERT -> REMOVE`
- `DELETE -> ADD`
- `UPDATE -> REPLACE`

并且统一按：

- `commit_lsn DESC`
- `record_lsn DESC`

排序。

如果最终结果“方向反了”或“顺序不对”，先看这里；如果只是某条 row 的 old/new 不对，不要先怪这里，先回到 replay。

## `fb_apply_keyed.c`

keyed 模式做的事：

1. 扫当前表，按 key 建 hash
2. 依 reverse stream 应用
3. 最后把 hash 中剩余 tuple 直接输出到结果 sink

几个关键实现点：

- key identity 是把 key 列序列化成字符串，不是直接拿 Datum hash
- 当前表扫描统一走 `fb_apply_scan_current_relation()`
- `REPLACE` 的处理是先移除 new key，再插入 old row

如果以后支持“主键变更后的更细粒度语义”，这一层很可能要继续改。

## `fb_apply_bag.c`

bag 模式的核心不是“按 ctid 回滚”，而是：

- 把 tuple 全量内容序列化成 `row_identity`
- 维护 `row -> count`
- `REMOVE` 就 `-1`
- `ADD` 就 `+1`
- `REPLACE` 就对 new 减一、old 加一

所以 bag 模式保证的是 multiset 语义，不保证物理重复行身份。

## 当前内存模型

query memory limit 现在不是 backend 全量 RSS 限制，而是扩展内部“已接入计费”的热路径限制。

当前已跟踪：

- `RecordRef` 数组
- payload arena 内的 `main_data / block data / FPI image`
- `BlockReplayStore`
- `ForwardOp`
- `ReverseOp`
- apply 阶段 keyed/bag 工作集

当前还未完全覆盖：

- TOAST live / retired chunk store 的全部增长
- 更大窗口下的 spool / spill / eviction 策略

所以遇到 OOM 风险时要区分：

- 是 `pg_flashback.memory_limit_kb` 主动报错
- 还是 backend RSS 实际超出系统承受范围

## 当前明确未完成的部分

截至当前代码：

- `fb_export_undo()` 仍未实现
- batch B / residual `missing FPI` 仍是主线 blocker
- TOAST store 全量内存硬上限还没完全覆盖
- 主键变更等剩余正确性还要补
- `parallel_segment_scan` 在真实主路径上的收益还需继续验证

所以你以后调 bug 时，先区分：

- 这是已知未完成边界
- 还是已宣称支持但当前实现有 bug

## 一句话抓住整个工程

这个项目现在的核心不是“把 WAL decode 出来”，而是“在只读前提下，用 checkpoint 之后足够完整的 WAL 为目标 relation 重建可信页状态，再把物理页变化提升成逻辑 reverse op，最终作用到当前表得到历史结果”。

只要你维护时始终按这条主线判断模块职责，就不容易走回旧的、错误的“直接拼 SQL / 直接拼 old tuple”路线。
