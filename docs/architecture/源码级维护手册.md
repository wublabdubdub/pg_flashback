# 源码级维护手册

本文面向接手当前主线代码的人，目标不是讲产品故事，而是让你在看源码时尽快建立正确心智模型。

读完后你应该能回答：

- 当前真正的查询主链是什么
- 代码里哪些模块是稳定边界，哪些还在收敛
- 出问题后先从哪一层查
- 修改时哪些旧模型绝不能带回来

## 一、先读顺序

按仓库当前约定，建议顺序是：

1. `AGENTS.md`
2. `STATUS.md`
3. `TODO.md`
4. `PROJECT.md`
5. `docs/specs/2026-03-22-pg-flashback-design.md`
6. `docs/architecture/overview.md`
7. `docs/architecture/核心入口源码导读.md`
8. `docs/architecture/wal-source-resolution.md`
9. `src/fb_entry.c`
10. `src/fb_wal.c`
11. `src/fb_replay.c`
12. `src/fb_reverse_ops.c`
13. `src/fb_apply.c`
14. `src/fb_apply_keyed.c`
15. `src/fb_apply_bag.c`
16. `src/fb_toast.c`

原因是：

- 当前主链已经不是“一个文件看完就懂”
- 入口、WAL、replay、apply 各自边界很清楚
- 但一些设计稿仍留在仓库里，必须先看 `STATUS/TODO` 才知道什么是已落地、什么是未完成

## 二、当前安装面与内部入口要分清

当前安装 SQL 只公开：

- `fb_version()`
- `fb_check_relation(regclass)`
- `pg_flashback(anyelement, text)`

安装脚本：

- `sql/pg_flashback--0.1.0.sql`

源码里还存在但不属于公开安装面的入口：

- `fb_archive_resolve_debug`
- `fb_recordref_debug`
- `fb_wal_sidecar_debug`
- `fb_export_undo`

其中：

- debug 入口主要被回归 SQL 临时创建包装函数使用
- `fb_export_undo` 还未实现完成

维护时不要把“源码里有函数”误认为“客户可直接调用”。

## 三、当前主链的真正边界

当前主链已经固定为：

```text
checkpoint + RecordRef + block redo + ForwardOp + ReverseOpSource + streaming apply
```

对应文件边界：

1. `src/fb_entry.c`
2. `src/fb_catalog.c` / `src/fb_guc.c`
3. `src/fb_wal.c`
4. `src/fb_replay.c`
5. `src/fb_reverse_ops.c`
6. `src/fb_apply.c`
7. `src/fb_apply_keyed.c` / `src/fb_apply_bag.c`
8. `src/fb_toast.c`

如果你看到任何“创建结果表”“返回结果表名”“并行结果表写入”的旧表述，默认都不是当前代码。

## 四、各模块职责和阅读重点

### 1. `src/fb_entry.c`

这里是统一入口层。

重点看：

- `fb_resolve_target_type_relid()`
- `fb_validate_flashback_target()`
- `fb_build_flashback_reverse_ops()`
- `pg_flashback()`
- `fb_flashback_cleanup_callback()`

你需要确认的点：

- 入口参数是怎么从 `NULL::schema.table` 变成 `relid`
- 为什么 progress 生命周期由入口层持有
- 为什么 SRF cleanup 不能省
- 为什么入口层现在只保留 `ValuePerCall`

### 2. `src/fb_catalog.c`

这里决定 relation 是否支持，以及 apply 走 `keyed` 还是 `bag`。

重点看：

- `fb_catalog_require_supported_relation()`
- `fb_catalog_find_stable_unique_key()`
- `fb_catalog_load_relation_info()`

维护时最容易出错的地方：

- 把“唯一但不稳定”的索引也当成 keyed
- 忽略 partitioned parent / system catalog / unlogged 的边界

### 3. `src/fb_guc.c`

这里已经不是简单 GUC 定义文件，而是当前来源解析策略的入口。

重点看：

- `_PG_init()`
- `fb_resolve_archive_dir_internal()`
- `fb_try_parse_archive_command_dir()`
- `fb_get_pg_wal_dir()`
- `fb_require_archive_dir()`

要点：

- `pg_flashback` 前缀已经 reserved，typo 不再静默接受
- `archive_dest` 是主语义
- `archive_dir` 只是兼容回退
- `archive_command` 只做安全可识别的本地自动发现

### 4. `src/fb_runtime.c`

文件很短，但它决定了很多运行时前提。

当前固定目录：

- `pg_flashback/runtime`
- `pg_flashback/recovered_wal`
- `pg_flashback/meta`

如果这些目录初始化逻辑被破坏，spool、ckwal、sidecar 都会连锁出问题。

### 5. `src/fb_wal.c`

这是当前最大、也是最容易误判状态的模块之一。

主要负责：

- resolver
- checkpoint / redo 锚点
- RecordRef 索引
- xid 状态
- unsafe window
- prefilter / tail inline / sidecar

阅读顺序建议：

1. 先看 `include/fb_wal.h`
2. 再看 `fb_wal_prepare_scan_context()`
3. 再看 `fb_wal_build_record_index()`
4. 最后再看 cursor、prefilter、sidecar 细节

原因：

- 如果一开始就扎进预筛选和缓存细节，很容易丢主线

### 6. `src/fb_ckwal.c`

这是 resolver 的延伸，不是一个完全独立的子系统。

你要记住：

- 它只负责复制/转换/回灌已有可用段
- 不是外部恢复工具封装
- 三处都没有可信段时必须直接失败

### 7. `src/fb_replay.c`

这是正确性最敏感的模块。

重点：

- `FbReplayStore`
- discover / warm / final
- block ready 检查
- missing block / missing FPI 收敛
- heap insert/delete/update/hot update redo
- TOAST page sync
- row image 提取

阅读时要特别区分两类逻辑：

1. 物理页恢复逻辑
2. 逻辑行变化生成逻辑

这两个逻辑现在在同一模块里，但职责不同。

### 8. `src/fb_reverse_ops.c`

这里是 replay 和 apply 之间的缓冲层。

重点：

- `FbReverseOpSource`
- run limit
- spill runs
- sort order
- reader

不要把它理解成“只是一个 vector”。
当前它已经承担了 bounded spill 进入主链后的关键边界。

### 9. `src/fb_apply.c`

这是 apply 协调器，不实现 keyed/bag 具体语义，但决定整体执行相位。

核心相位：

- `FB_APPLY_PHASE_SCAN`
- `FB_APPLY_PHASE_RESIDUAL`
- `FB_APPLY_PHASE_DONE`

重点：

- 为什么扫描当前表时不再把整表装进内存
- 为什么 residual 阶段单独作为 stage 9
- 为什么 materialized 与 value-per-call 共用一套 apply 状态

### 10. `src/fb_apply_keyed.c`

当前 keyed 不是“按整表构 hash”，而是“只维护变化 key”。

重点：

- `FbKeyedApplyState`
- `FbKeyedEntry`
- typed key fast path
- `replacement_tuple`
- `current_seen`

维护时最重要的两个问题：

1. 同 key 多次变化如何收敛
2. 主键变化场景是否会把 old/new key 混掉

### 11. `src/fb_apply_bag.c`

bag 的核心不是 tuple 身份，而是 `delta`。

重点：

- `FbBagEntry.delta`
- `residual_repeat`
- typed row identity fast path

维护时要牢记：

- bag 只保证内容与重复次数
- 不保证同值重复行的物理身份

### 12. `src/fb_toast.c`

这里不是“辅助工具模块”，而是当前正确性主链的一部分。

重点：

- `FbToastStore`
- `live_chunks`
- `retired_chunks`
- spill retired chunk
- `fb_toast_store_sync_page()`
- `fb_toast_rewrite_tuple()`

维护时必须记住两条规则：

1. 主表行像要优先从历史 TOAST store 重建，不读 live 错值
2. 如果历史 chunk 真缺失，必须报错，不能偷偷返回不完整值

## 五、当前关键结构体该怎么记

### `FbRelationInfo`

定义：

- `include/fb_common.h`

它回答三个问题：

1. 目标表是谁
2. 有没有 TOAST
3. 走 keyed 还是 bag

### `FbWalScanContext`

定义：

- `include/fb_wal.h`

它回答：

- 从哪里扫
- 扫到哪里
- 锚点在哪里
- 本轮来源解析发生了什么

### `FbWalRecordIndex`

定义：

- `include/fb_wal.h`

它回答：

- 本轮要 replay 哪些 record
- 事务状态是什么
- 有没有 unsafe window

### `FbReverseOpSource`

定义：

- `include/fb_reverse_ops.h`
- 实现在 `src/fb_reverse_ops.c`

它回答：

- replay 产生的逻辑变化，apply 应该按什么顺序消费
- 变化量太大时如何 spill

### `FbApplyContext`

私有定义：

- `src/fb_apply.c`

它回答：

- 当前表扫描进行到哪
- residual 还有多少
- keyed/bag 子状态在哪里

## 六、当前小内存边界到底落在哪

### 已经解决

1. keyed 不按当前整表建立 hash
2. bag 不按当前整表建立 multiset
3. reverse-op 可以 spill runs

### 还没完全解决

1. `RecordRef` payload
2. replay block store
3. page image / block data / main data 高水位

所以如果你看到大 live case 还需要较高 `memory_limit`，不要先怀疑 apply；当前瓶颈更多还在 WAL/replay 前半段。

## 七、当前最常见的误判

### 1. 以为 `pg_flashback()` 一定是 value-per-call

准确。
当前默认语义就是直接查询型 `ValuePerCall` SRF，内部 materialized SRF 分支已移除。

### 2. 以为 TOAST truncate 仍然是 relation 级必拦

不准确。
主表仍严格拦，TOAST truncate 当前已放宽为“允许 replay，缺 chunk 再报错”。

### 3. 以为 aborted record 可以直接跳过

不准确。
逻辑上不进入最终结果，不等于物理页 replay 可以跳过。

### 4. 以为 `archive_dir` 仍是主配置

不准确。
当前主配置语义已经是 `archive_dest`。

### 5. 以为 `pg_flashback-summaryd` 已经默认不连数据库

不准确。
当前 daemon 只是把 summary 服务从 preload/bgwoker 形态外移，
build / cleanup 仍通过 libpq 连接扩展内部 debug helper 完成。

所以当前真实启动方式应显式带：

```bash
pg_flashback-summaryd \
  --pgdata /path/to/pgdata \
  --archive-dest /path/to/archive \
  --conninfo "host=/tmp port=5432 dbname=postgres user=postgres connect_timeout=2" \
  --foreground
```

如果省略 `--conninfo`，经常会因为 libpq 默认 `dbname/user`
不匹配而直接失败。

## 八、修改代码时的硬约束

1. 不要把旧结果表接口重新加回安装面
2. 不要重新引入 `fb_parallel` / `parallel_apply_workers`
3. 架构或接口变更前先更新 `STATUS.md` / `TODO.md` / 必要时 ADR
4. 改了 `include/*.h` 后，不要只做增量安装

统一做：

```bash
make clean
make install
```

## 九、当前真正需要继续补的主线

维护视角下，当前最重要的未完成项仍然是：

- batch B / residual `missing FPI`
- WAL 索引 / replay 继续 bounded spill 化
- `fb_export_undo`
- 主键变化与更多 heap WAL 正确性
- 宽表 / TOAST 极端场景验证

## 十、最短结论

当前源码维护的关键不是“多记几个函数名”，而是始终分清这四层：

```text
WAL 负责提纯事实
replay 负责恢复页和行像
reverse-op 负责表达历史逆向变化
apply 负责把这些变化映射成最终结果集
```
