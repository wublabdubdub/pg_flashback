## 当前安装面

当前安装 SQL 只暴露：

- `fb_version()`
- `fb_check_relation(regclass)`
- `pg_flashback(anyelement, text)`

当前真实用户调用形态：

```sql
SELECT *
FROM pg_flashback(
  NULL::public.t1,
  '2026-03-23 08:09:30+08'
);
```

不再存在：

- `pg_flashback(text, text, text)` 结果表入口
- `fb_parallel` 并行写结果表路径
- `parallel_apply_workers`
- `fb_flashback_materialize(...)` 公开安装面

## 主链分层

### 1. 入口层：`src/fb_entry.c`

负责：

- 解析复合类型锚点
- 解析目标时间文本
- 进入 relation / runtime gate
- 串起 WAL / replay / reverse-op / apply
- 以 value-per-call SRF 逐行返回结果

### 2. relation gate：`src/fb_catalog.c`

负责：

- relation 类型校验
- 选择 `keyed` 或 `bag`
- 记录 TOAST relation 信息

### 3. WAL 层：`src/fb_wal.c`

负责：

- WAL 来源解析
- checkpoint 锚点搜索
- 时间窗扫描
- 目标 relation `RecordRef` 索引
- unsafe window 判定

### 4. replay 层：`src/fb_replay.c`

负责：

- `BlockReplayStore`
- `checkpoint + FPI + block redo`
- 从重放前/后页面提取 `ForwardOp`

### 5. reverse-op 层：`src/fb_reverse_ops.c`

负责：

- `ForwardOp -> ReverseOp`
- 反向操作流排序

### 6. apply 层：`src/fb_apply.c` + `src/fb_apply_keyed.c` + `src/fb_apply_bag.c`

负责：

- 流式扫描当前表
- 对变化集执行 reverse apply
- 逐行返回当前行或 residual 历史行

### 7. TOAST 层：`src/fb_toast.c`

负责：

- 历史 TOAST chunk store
- external toast datum 重建
- row image inline 化

## 当前关键结构体

### `FbRelationInfo`

定义于 `include/fb_common.h`。

保存：

- `relid`
- `toast_relid`
- `mode`
- `key_attnums`

它决定：

- 目标表能不能查
- 走 `keyed` 还是 `bag`
- 是否需要 TOAST 辅助链路

### `FbWalScanContext`

定义于 `include/fb_wal.h`。

保存：

- `target_ts`
- `query_now_ts`
- anchor checkpoint / redo
- WAL 来源解析结果
- unsafe 标记

### `FbWalRecordIndex`

定义于 `include/fb_wal.h`。

保存目标 relation 在时间窗内需要 replay 的结构化 WAL 引用。

### `FbForwardOpStream` / `FbReverseOpStream`

定义于 `include/fb_reverse_ops.h`。

它们是 replay 层与 apply 层之间的主要边界。

### `FbApplyContext`

定义于 `src/fb_apply.c` 私有实现。

保存：

- 当前 relation scan 状态
- 当前阶段（scan / residual）
- keyed / bag mode state

## 当前结果模型

当前结果模型已经固定为：

- 不创建结果表
- 不返回结果表名
- 不做最终落盘
- 通过 SRF 按行返回

这意味着：

- 用户的一次查询就是一次完整 flashback 计算
- 如果同一历史结果需要被多次复用，应该由调用方自己在 SQL 层决定是否另行物化

## 当前内存边界

### apply 层已解决的问题

- 不再把当前整表装进 keyed hash
- 不再把当前整表装进 bag multiset
- 12GB 当前表不会仅因 apply 而需要 12GB 工作集

### 仍需继续推进的边界

- `RecordRef` 数组
- FPI / block data / main data
- replay state

这些仍受 `pg_flashback.memory_limit_kb` 约束；超过即报错。

## 代码修改时的注意点

- 若改了 `include/*.h`，统一执行 `make clean && make install`
- 不要把“旧结果表接口”重新加回公开安装面
- 不要重新引入 `fb_parallel` / `parallel_apply_workers`
- 若改动接口或架构，先更新 `STATUS.md` / `TODO.md` / 相关 ADR

## 当前调试入口

对外仍可直接用的只有：

- `fb_version()`
